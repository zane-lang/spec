# Stories: Types

> **See also:** [`spec/types.md`](../spec/types.md) — the rules these chapters explain.

## Two axes instead of a keyword per kind

A type in Zane answers two independent questions, and the design turns on refusing to fuse them. The first is *shape*: is this a product (fields side by side) or a sum (one of several cases)? The second is *kind*: is this a value — copied, identity-less, transitively value — or a reference — a single-owner thing with stable identity that other things can point at? Those are orthogonal. A product can be either kind; so can a sum. Four cells in a two-by-two grid.

The mainstream move is to spend a keyword on each cell, or on most of them: `struct` for the value product, `class` for the reference product, `enum` or `union` for sums, each a separate declaration form with its own rules bolted on. We rejected that, because it makes a reader learn four things where there are really two, and it hides the fact that value-versus-reference is the *same* distinction whether you are looking at a product or a sum. If `class` is "the reference one" for products, nothing in `class` tells you what "the reference one" would be for a sum — the knowledge doesn't transfer, because the grammar split it across unrelated keywords.

So we kept the two axes literally two axes. One keyword picks the shape — `struct` for a product, `variant` for a sum — and one modifier, `#`, picks the kind: unmarked is a value type, `#`-marked is a reference type. `struct`/`#struct` are the product pair, `variant`/`#variant` the sum pair, and the `#` means the same thing in both places ([`types.md` §2.1](https://github.com/zane-lang/spec/blob/69cd84780e2f5979c009243ce65d6d776c167cb5/spec/types.md#21-the-valuereference-axis-and-the--modifier)). Learn `#` once and it composes across the whole grid. The value type is transitively value and mutable in place; the reference type carries identity and may hold `&` fields and recurse ([`types.md` §2.2](https://github.com/zane-lang/spec/blob/69cd84780e2f5979c009243ce65d6d776c167cb5/spec/types.md#22-value-types-are-transitive-and-mutable-in-place)) — but that behavioural split rides on the one modifier rather than on a proliferation of keywords.

The cost is subtle and it takes the rest of this file to pay off. Writing the kind as a *modifier* invites reading it as an annotation — something you can staple onto any type expression the way you'd staple `const` or `mut`. `#struct { … }` reads as "the struct, but reference"; and if that is how `#` works, then `#Int` reads as "the `Int`, but reference", and `#T` as "the `T`, but reference", for any `T` at all. The first draft of the spec took exactly that reading and let `#` compose with every type expression, primitives included. Undoing that over-generous reading is the through-line of the last two chapters.

## What `#` actually changes, and the boxing trap

Before deciding where `#` may appear, we had to be honest about what it *does*, because the intuitive answer is wrong and the wrong answer quietly justifies the over-generous grammar. Ask what the difference is between `Int` and `#Int` and the reflex is: `#Int` is boxed — a heap allocation, a pointer to an `Int` instead of an `Int`. It *feels* boxed, because you can hold an `&#Int` and hand it around, and a thing you point at surely lives behind a pointer.

That reflex is false, and the spec goes out of its way to make it false. Placement — stack or heap — is an unobservable implementation choice; a reference-type instance is put on the heap only when its size is dynamic or it is moved into a heap owner, and moving it into a longer-lived stack slot is just a relocation ([`memory.md` §3.5](https://github.com/zane-lang/spec/blob/69cd84780e2f5979c009243ce65d6d776c167cb5/spec/memory.md#35-reference-type-instances-may-be-placed-on-the-stack)). So "`#` means boxed" cannot be the distinction: two programs that differ only in where the compiler placed a `#Int` are the same program. What `#` actually adds is *identity* — an anchor ([`memory.md` §4.1](https://github.com/zane-lang/spec/blob/69cd84780e2f5979c009243ce65d6d776c167cb5/spec/memory.md#41-the-anchor-table)), single ownership, and the ability to be aliased through `&`. The `&#Int` you can hold is not evidence of indirection; it is evidence of an anchor. The pointer feeling is real, but it is the anchor talking, not a box.

Getting this straight reframes the whole kind axis. `#T` is not `T`-with-a-box-around-it, a representation trick over the same type. It is a **different type** that happens to share `T`'s *underlying architecture* — its field layout — and nothing else. It has its own identity discipline, and because it is a distinct type it has its own constructors, its own operators, its own methods. The value `Int` and the reference `#Int` are as unrelated, behaviourally, as any two nominal types that happen to have the same fields.

The cost of taking that seriously is that there is no free behavioural reuse. If `#Int` is a genuinely separate type, a method or operator written for `Int` does not apply to it; only the structure carries over, and behaviour has to be declared again. That is a real price — it is the reason the temptation to treat `#` as "same type, just boxed" never fully dies — but it is the honest consequence of the identity story, and paying it is what lets the next chapter talk about constructing a `#` type without incoherence.

## Naming a type by what it is, not how it is built

Once `#T` is its own type, the immediate question is how you construct one, and that question runs straight into a decision already made for generics: there is no turbofish. A call carries no `<>` list; a type reaches a constructor by inference off a value argument or as an explicit `Type` value, never as an angle-bracket channel at the call site (the [generics story's no-turbofish chapter](generics.md#no-turbofish-passing-types-as-values) tells why, and the rule lives in [`generics.md` §5](https://github.com/zane-lang/spec/blob/69cd84780e2f5979c009243ce65d6d776c167cb5/spec/generics.md#5-constructor-and-function-calls)). The principle underneath it is worth stating in its own terms: **a constructor is called by what the type *is*, not by the architecture it is built from.** `Vector<Int>` names an architecture — `Vector` applied to the parameter `Int` — and a use site does not respell that architecture; it writes `Vector(…)` and lets the `Int` ride in on the arguments.

Read that way, letting `#name` be a constructor looks not just allowed but *right*. The objection to turbofish is that `<>` smuggles a type's *parameters* — its architecture — into the value channel, redundantly with the inference that already carries them. But `#` is not a parameter. Nothing on the arguments reveals it: an `Int(10)` argument cannot tell you whether the caller wanted a value or a cell, so `#` is not a redundant second channel for something inference already has — it is intrinsic to *what the type is*. By the very principle that kills turbofish, `#Int(10)` — naming the type you are constructing — is the thing the rule wants, not the thing it forbids. We spent real time on this road, and it is a genuinely nice library-API trick: `#T` as a first-class type you construct by name.

It wobbled on the use-site consequences. Making `#name` a constructor means `#` now appears in call position, which means `#` is a thing that can appear wherever a type is *used*, not only where one is declared — and that reopens everything the first chapter's "modifier" framing threatened. In generic code a parameter `T` is sometimes a value type and sometimes a reference type, and the moment you can write `#T` you are forming reference types out of parameters and mixing `#T` with `#U` and with a bare `V` that is *already* a reference type, and asking what `#T` even means when `T` is already `#`-marked. The trick that was clean for `#Int` at a concrete call site turned into a swamp the instant `T` was a variable. The tension is real and it is left open at the end of this chapter; the next one resolves it by attacking the premise — that `#` should be a use-site thing at all.

## Confining `#` to the body forms

The swamp came entirely from `#` being applicable to arbitrary type expressions — primitives, named types, parameters. So the resolution is to take that away: `#` may mark only the three **body forms** — `#struct`, `#variant`, `#enum` — and a body form only ever appears at a *declaration*. There is no bare `#Int`, no `#T`, no `#Vector<Int>`; the current spec's permission for `#` to compose with a primitive or a named type ([`syntax.md` §2.10](https://github.com/zane-lang/spec/blob/69cd84780e2f5979c009243ce65d6d776c167cb5/spec/syntax.md#210-the--reference-modifier)) is exactly what gets withdrawn. `#` stops being a modifier you apply and becomes part of how you *declare* a reference type.

The consequences cascade, and they are almost all the problems of the previous two chapters dissolving at once. Every reference type is now declared, so every reference type has a name and therefore a home package — which answers, without a new rule, the question of who may hang constructors, operators, and methods on it. Because the reference type is named, its constructor is the bare name — `Node(…)`, `Weapon(…)` — and the `#` never reaches a call site. That means the `#name`-constructor idea is not so much rejected as made unnecessary: there is no anonymous `#T` left that would need a `#`-bearing constructor, so the whole turbofish-adjacent tension about `#` in call position simply has nothing to attach to. And the generics swamp drains: `T` can still *be* a reference type — you can instantiate a generic with `Node` — but you can no longer *form* `#T`, so the double-`#`, the `#T`-mixed-with-`#U`, the "is this parameter already boxed" questions cease to be expressible. The confusion was never inherent to polymorphism; it was manufactured by a use-site `#`, and deleting the operator deletes it.

The price is paid in expressiveness, and it is not nothing. You lose the ad-hoc cell: there is no `&#Int` for a shared, aliasable plain integer; if you want one you must declare a named cell type — `type Counter = #struct { n Int; }` — and share a `&Counter`. You lose use-site `#`-flipping: given a value type you cannot summon its reference counterpart at the point of use, you re-declare the body, which duplicates layout wherever you want both kinds. And there is no generic "reference-ise this `T`" at all. For a language whose philosophy already holds that value and reference are separate types sharing only structure, making the user *name* the reference type they want is arguably the honest tax rather than a loss — a shared mutable cell is a concept, and concepts earn names — but it is a real tax and worth naming as one.

One edge is still open where this chapter stops. It is whether an *anonymous* inline `#struct` — a reference body that appears as a member's type without a `type` declaration to name it — should be allowed at all, or whether reference body forms must always be named so that the "every reference type has a name and a home" property holds without exception. The clean answer is to require the name, but it is a restriction on inline type expressions that needs its own justification, so it is left for the chapter that resolves it; recording it here is the point, so the next reader does not mistake an unfinished edge for a settled rule.

## Every type has a name, because construction needs one

The edge chapter 4 left open — whether an anonymous inline `#struct` must be named — turned out to be the visible corner of a rule bigger and simpler than the question that surfaced it. The honest answer is not "reference types must be named" but *every* type must be: a direct inline type, written where a type is used rather than declared, is illegal outright. A field, parameter, or return type may only **name** a type — a bare name, or a named type applied to arguments — never spell a fresh one out in place.

What forces this is construction, the same package-scope, name-based construction the rest of this file already leans on ([`types.md` §3.1](https://github.com/zane-lang/spec/blob/69cd84780e2f5979c009243ce65d6d776c167cb5/spec/types.md#31-constructors-are-package-scope-declarations)). You build a value by writing its type's name and an argument list; the constructor is a verb named after the type. An anonymous type has no name, so it has no constructor, so a value of it cannot be built the way every other value is — you would need a second, nameless construction syntax beside the first, and a type that can sit in a field but that you cannot refer to anywhere else. That is not an awkwardness at the edges; it is a type sitting outside the construction model. Rather than invent a parallel way to build the nameless ones, we removed the nameless ones. If a shape is worth using, it is worth a `type` declaration; once it has that, it is an ordinary named type with an ordinary constructor.

We reached for a narrower line first — ban the inline *bodies* (`struct`, `variant`, and their `#` forms) but keep the structural type expressions inline, on the grounds that those merely apply an existing builder rather than define a new type ([`syntax.md` §2.4](https://github.com/zane-lang/spec/blob/69cd84780e2f5979c009243ce65d6d776c167cb5/spec/syntax.md#24-type-expressions) currently permits both). It reads well until you follow it into construction: an applied generic like `Vector<Int>` is built by name, but a bare structural shape is not, and drawing the legal line anywhere other than "does this have a name-based constructor" leaves a use site that sometimes names its type and sometimes spells it out. One kind of type at a use site — a named one — is the property worth having, and the only line that delivers it cleanly is the total one. The `init{ }` construction body is so far specified only for structs; how a constructor builds a variant, or any non-struct form, is not yet settled, so this rests on the principle rather than on a particular construction form. However those bodies land, the types they build are reached through a `type` declaration, not spelled out inline.

The cost is namespace pressure, and it is real. A one-off grouping that a nested language would tuck inline now needs a top-level `type` and a name, so a cluster of small shapes becomes a cluster of package-scope names, and the "just group these two fields here" convenience is gone. We take that as the honest tax of a single construction model rather than a regression: a shape common enough to embed is usually a concept worth naming, and the reader who meets the field gets a name to look up instead of a body to parse. Whether to soften the tax with block- or section-local `type` declarations — named and nominal, but not raised to the package namespace — is left open; it would keep every type named while letting the name be local, and nothing here forecloses it.

## Coercion follows the call, not the store

The last chapters settled how a value gets *built* — [by name, through a constructor](#naming-a-type-by-what-it-is-not-how-it-is-built). This one is about when the compiler builds one *for* you: an implicit constructor, inserted silently so that a `Feet` can satisfy a slot asking for `Meters`. The machinery itself was never in doubt; where it may fire was, and the pressure that forced the question was plain verbosity. A positional constructor argument coerced — `printDistance(Feet(…))` quietly became `printDistance(Meters(Feet(…)))` — but the field-constructor form did not. So `Point{x = 3, y = 4}` was rejected where `Point(3, 4)` was accepted, and you were made to write `Point{x = Int(3), y = Int(4)}`, spelling out a wrap the paren form spared you. Two syntaxes for the same act — calling a constructor — disagreeing on whether the compiler would meet you halfway.

The first instinct was to read this as a missing *feature* and reach for a large one: target-typed construction, where a bare `{x = 3, y = 4}` infers its type from the slot it lands in. That road is real and other languages walk it, but it was the wrong diagnosis — it answers "construct without naming the type," and the actual complaint was narrower and sharper. A field-constructor call already names its type; it simply was not being treated as the *call* it plainly is. Once the exclusion was seen for what it was — an inconsistency, not an absent feature — the fix shrank to a single move: make the named field entries of a `Type{ … }` call coercion sites too, exactly like the positional arguments of a `Type(…)` call ([`types.md` §4.2](https://github.com/zane-lang/spec/blob/ae1fddfa71a138e16699890d253042030b77b3fc/spec/types.md#42-coercion-sites)). The `()`/`{}` disagreement dissolves because both are now the same kind of thing.

The harder question was where to *stop*, and we did not get it right on the first pass. The obvious next candidates were the other places a value meets a declared type: the `init{ }` a constructor body writes its fields through, a `return`, a plain assignment, a declaration. We took `init{ }` in first, on a tidy syntactic rule — *every* `{ field = expr }` coerces, whether it is a call at a use site or an `init` in a body — and it lasted about as long as it took to say out loud. Because `init{ }` is not the caller satisfying someone's contract; it is the constructor *authoring its own value's fields*, in the very type definition where those field types are written. That is the same posture as a `return`, which hands a value to the return type *you* declared, and as an assignment, which stores into storage whose type was fixed at its declaration. Pulling `init` back out was the moment the real line showed itself.

The line is not "wherever the destination type is known" — it is known in every one of these places. It is *whose* type it is. A coercion fires where a value is passed into a **call or constructor**, adapting to a contract fixed by *someone else's* signature; it does not fire where the value is written into a type *you* declared locally — a symbol declaration, an assignment or store, a `return`, or the fields your own constructor builds through `init{ }` ([`types.md` §4.2](https://github.com/zane-lang/spec/blob/ae1fddfa71a138e16699890d253042030b77b3fc/spec/types.md#42-coercion-sites)). Field-constructor call entries sit on the first side, and that is the whole of the change; `init`, `return`, assignment, and declaration sit on the second, and stay explicit. One test, applied honestly, sorts every position.

`match` is the satisfying check on that test, because it needed no new rule at all. The scrutinee is never coerced — you dispatch on the value's live tag, so you must already hold the variant, for the same reason a method receiver is never converted ([`types.md` §4.6](https://github.com/zane-lang/spec/blob/ae1fddfa71a138e16699890d253042030b77b3fc/spec/types.md#46-method-receivers-are-never-implicitly-converted)). And an arm body is a `return` in disguise: `=> expr` is shorthand for `{ return expr }` ([`adt.md` §5.1](https://github.com/zane-lang/spec/blob/ae1fddfa71a138e16699890d253042030b77b3fc/spec/adt.md#51-the-block-and-its-arms)), so an arm whose value differs from the match's result type must wrap it explicitly, precisely as any other return does. Coercion still fires *inside* an arm, at the ordinary call boundaries there — but the arm-to-result handoff is the return side, and stays explicit. A construct we never once mentioned while drawing the line already obeys it.

The price is a papercut you can see. `Point{x = f}` at a call coerces, and `init{ x = f }` in the body beside it does not, though the two read almost identically; a reader has to know that the type name in front of the brace is what makes one a call and the other a store, and that is a fine distinction to hang a behavioural difference on. We think it is the right papercut — it is the visible edge of a line worth having, and the alternative was a syntactic rule that was easier to state and wrong — but it is a real thing to learn, and pretending the two braces look different would be dishonest. The other unpaid bill is the literal that started all this: `{ val = 3 }` only loses its wrap if the core numeric types expose their concept-to-`Int` constructors as `implicit`, and whether a bare literal should lower against any locally-known type is a separate lever we deliberately left on the bench rather than solve under cover of this change.

## Naming the moulds, and marking every one

[Confining `#` to the body forms](#confining--to-the-body-forms) left the language with four constructs that give a type its shape — `struct`, `variant`, `enum`, and `tuple` — and no single word for the group. We had been calling three of them the "body forms," but that label was doing sly work: it never meant "the forms with a body," it meant "the three the `#` mark was confined to," and it quietly shut `tuple` out while lumping `enum` in, even though an `enum` is a bracketed `[ ]` list exactly like a `tuple` and nothing like the `{ }` bodies of `struct` and `variant`. The set the label named was real; the reason behind its membership was not the one the name implied.

The pressure that forced the issue was mundane: we needed to *talk* about the four together — in the spec, and in a compiler's syntax tree — and a group used on every other page deserves a name. We went looking for one and rejected a few. "Origin form" was accurate — every type, the core types included, is cast from one of the four — but it states the definition instead of lending an image, and a name is not a summary. "Matrix" was the buried gem: in typography a *matrix* is the mould a piece of metal *type* is struck in, so it ties straight to the word "type" the language already uses — but a programmer already holds "matrix" for a math object and a container, and a term that has to fight its own reader every time it appears is a bad term however clever the etymology. "Arche," the Greek for a first principle, was too exotic to read naturally on the third line of a spec paragraph. We landed on **mould**: it keeps the exact casting metaphor "matrix" carried — the form a type is cast from — with none of the baggage, and it pairs with the existing `verb` in register, so the language now has a plain word for the things that *shape* data beside the plain word for the things that *do* work.

Naming the four as peers is what exposed the `tuple` gap as the accident it was. If value-versus-reference is genuinely one axis orthogonal to shape — the claim the whole two-axis model rests on ([`types.md` §2.1](https://github.com/zane-lang/spec/blob/b350ce8cc845236c3cd116504915014fb4c6f661/spec/types.md#21-the-valuereference-axis-and-the--modifier)) — then every shape should cross it, and singling `tuple` out is a dent in the very orthogonality that makes the single `#` modifier elegant. We looked hard for a real reason to keep `tuple` value-only, and the best one is a nudge, not a wall: a reference type has identity and is aliased through `&`, and the moment a product is important enough to carry identity its fields are arguably important enough to *name*, so a reference product ought to be spelled `#struct` and `#tuple` is a positional, worse-read spelling of the same thing. That is a genuine ergonomic argument. But it is style advice, and we were letting it masquerade as a grammar rule — while the grammar already contradicted it, since `#enum` exists and an `enum` is a bracketed list no more "bodied" than a `tuple`. So we removed the carve-out: `#` marks any mould, `#tuple` included, and a reference tuple is just a positional reference product placed by the ordinary reference-type rules ([`memory.md` §3.5](https://github.com/zane-lang/spec/blob/b350ce8cc845236c3cd116504915014fb4c6f661/spec/memory.md#35-reference-type-instances-may-be-placed-on-the-stack)). With every mould now taking the mark, "body forms" and "mould" named the same set, and the older label — always the weaker one — was retired.

The cost is the nudge we gave up. Nothing now steers a writer away from `#tuple[Int, &Self]` toward the `#struct { head Int; tail &Self }` that a reader would usually thank them for, so the language permits a positional reference type that is rarely the best choice, and "name your fields once a product has identity" drops from an enforced line to advice a style guide has to carry. We think that is the honest trade: buying back the orthogonality of the two-axis model — one mark, every shape, no exceptions to memorize — is worth more than a forcing function the grammar was enforcing by accident and only three-quarters of the way.

## The tuple was a product we already had

The [previous chapter](#naming-the-moulds-and-marking-every-one) talked itself into a corner it did not quite see. It had spent its length arguing `tuple` into full peerage — `#` marks *every* mould, `tuple` included, for an orthogonality the two-axis model wanted to keep clean — and then, in its last paragraph, named the price with unusual candor: nothing now steers a writer away from `#tuple[Int, &Self]` toward the `#struct { head Int; tail &Self }` a reader would thank them for. We had written down, in our own words, that the feature we were polishing was one we would advise people not to use. That is not a cost a good feature carries; it is a feature confessing it should not exist.

What forced the reckoning was specifying construction. When we finally pinned down how a variant is built — [naming a case](adt.md#naming-a-case-not-calling-a-constructor) — we were working from the rule that runs under the whole type system: [every constructible type is named](https://github.com/zane-lang/spec/blob/4509edd42959886a8cc539f6e0a211da17f771b6/spec/types.md#53-the-right-hand-side-is-a-type-expression), and construction is what a name is *for*. Held against that rule, `tuple` was a hole, not a mould. It had a declaration form and nothing else: no literal to build one, no positional access to read one back, and the single consumer that could have given a product-of-values any meaning — `match` — [refuses on principle to destructure it](https://github.com/zane-lang/spec/blob/4509edd42959886a8cc539f6e0a211da17f771b6/spec/adt.md#53-variant-matching-not-pattern-matching). Every other mould earned its keep by what you could *do* with a value of it; `tuple` earned its keep by being a shape in a table. Strip the orthogonality argument away and what remained was a positional product — and a `struct` is a product with its fields named, the same thing done better in every case a reader has to live with afterward.

We looked at finishing it rather than cutting it. Give `tuple` a literal and an access syntax — positional `.0`, `.1`, or the 1-based counting the rest of the language uses — and it becomes a usable type. But that walks straight back into the ergonomics the previous chapter had already called worse, now with more surface to specify, and it would have to reconcile positional field access with the no-destructure stance the multi-scrutinee rule leans its whole weight on. Keeping it inert — nameable but unbuildable — was the other option, and it fails the naming rule outright: an unconstructable mould is exactly the dead corner "every constructible type is named" exists to forbid. So we dropped it. There are three moulds now — `struct`, `variant`, `enum` — and the one place the spec had used a tuple, a `QualifiedIdent` of two strings, is a two-field `struct` that reads better than the pair it replaced, which is the whole argument in miniature. Nothing rhetorical is lost: the multi-scrutinee block still refuses parentheses because they would name a tuple to take apart, and you need no mould in the language to reject a concept — Rust's `Result` is named in the spec the same way, as a foil, without being part of Zane.

The cost is the orthogonality we had bought one chapter earlier and now hand back. "One mark, every shape, no exceptions to memorize" was a real elegance, and the mould set is no longer the tidy cross-product of {value, reference} against every shape — it is three shapes that each cross the `#` axis, with a fourth shape simply not there. But an orthogonality maintained across a shape we would tell no one to use is symmetry for its own sake, and paying for it with a whole construct — one more keyword, one more entry in every mould list and summary, one more thing a reader of the grammar must hold — is a bad trade we had made mostly because the symmetry looked clean on the page. Fewer moulds, each pulling its weight, is the honest shape of it. What we truly give up is the terse positional product for the rare moment someone wants a bare pair and would rather not name its halves; they write the `struct` and name them, a few more characters and a better line for whoever reads it next.

## Named constructors, and the syntax variants already had

A type usually has more than one way to be made. A `Vector2` is built from its `x` and `y`, but it is also, meaningfully, "the origin," or "a diagonal of length *n*," or "the point at polar `(radius, angle)`" — and the anonymous by-type-name constructor can tell those apart only by the *shape* of their arguments. Overloading carries you a little way: `Vector2(Float, Float)` and `Vector2(Float)` are distinct because their arities differ. But two paths of the *same* shape — `fromCartesian(x, y)` and `fromPolar(radius, angle)`, both two `Float`s — collide, and even where arities differ, the call site `Vector2(Float(0), Float(0))` says only "two floats," never "the origin." The intent that lives in the *name* of a construction path had nowhere to go.

Dart's answer is named constructors — `ClassName.named(...)` — and we took it because that pressure is real and the addition costs almost nothing. The idea had been logged as [#124](https://github.com/zane-lang/spec/issues/124), but a standing request is no justification on its own; what earns named constructors their place is the construction paths overloading cannot tell apart. Naming a verb after a type was already the marker that made it a constructor (implicit return, `init{ }` unlocked); all a named constructor does is let that name carry a `.suffix`, the same two unlocks with the suffix only distinguishing which path this is. We rejected the alternatives quickly: a separate `static`-method or `new`-keyword syntax would mint grammar for something the marker model already covers, and Zane spends its budget avoiding exactly that; keeping overloading-only would leave the same-shape paths permanently unnameable. The casing rule does the disambiguation for free — `Vector2.zeros` has an uppercase receiver, so `.zeros` is a member of the *type* and reads as a constructor, while `v.zeros` is a member of a *value* and reads as a field or method. Nothing new had to be taught.

What made the addition land *now* rather than as its own separate feature is that variants had, one design over, just introduced the exact surface a named constructor wants. Building a variant is [naming a case](adt.md#naming-a-case-not-calling-a-constructor): `Expr.intLit("5")`, a `Type.member(args)` form, in both the long `e Expr = Expr.intLit("5")` and the short `e Expr.intLit("5")` declaration. A named struct constructor is spelled identically — `v Vector2.diagonal(Float(3))`, long or short — so instead of designing a second syntax we reused the one already on the page. That is the whole of the connection, and its size matters, because the tidy overclaim is right there and it is wrong. It is tempting to say a variant case *is* the built-in named constructor of a sum, one mechanism in two hats. It is not. A [named constructor](https://github.com/zane-lang/spec/blob/379542642df44bf3e9c0206efd8e1e5bcaeff228/spec/types.md#34-named-constructors) is a *verb* — you declare it, give it a body that runs, overload it, and it builds through `init{ }`. [Naming a variant case](https://github.com/zane-lang/spec/blob/379542642df44bf3e9c0206efd8e1e5bcaeff228/spec/adt.md#32-constructing-a-variant) is built-in syntax with no verb behind it: nothing is declared, nothing generated, the case names itself. They share a spelling and diverge underneath, so the record should say *rhymes with*, never *unifies* — the syntax is one, the machinery is two.

The cost is small and we will name it anyway. There are now two spellings for "make a `Vector2` a particular way" — an anonymous overload told apart by argument shape, or a named constructor told apart by its name — and the grammar does not choose between them; a writer does. When two paths have distinct shapes, either works, and "name it or overload it?" becomes a style call the language declines to make for you. We think that latitude beats a rule forcing one or the other, but it is real, and it is the kind of thing a house style ends up settling rather than the compiler.

## Naming the last shape: the peer mould

Dropping `tuple` left a smaller loose end than the feature itself, and we did not see it until the set had shrunk. Of the three moulds now remaining, two were named by their shape — the product `struct`, the sum `variant` — and the third, `enum`, had only ever been "the other one." While `tuple` was also unnamed the gap was invisible; once `enum` was the *lone* shape without a name of its own, the asymmetry was conspicuous, and a category the spec talks about on every ADT page deserved better than a placeholder.

That it is a real category, and not just an underspecified sum, is a point worth making carefully, because the tidy dismissal is wrong. A [`variant`'s](https://github.com/zane-lang/spec/blob/71c25fe4ee7f756f03d0a420483f37a1df622350/spec/adt.md#3-variants) representations count as the sum of its cases' *payload* cardinalities; an [`enum`'s](https://github.com/zane-lang/spec/blob/71c25fe4ee7f756f03d0a420483f37a1df622350/spec/adt.md#2-enums) members carry no payload at all, so its count is simply the number of members. You can call that a degenerate sum — a sum of units, 1+1+…+1 — and be technically right, but the degeneracy *is* the distinction: a shape whose members have no interior is a different thing to reason about, tabulate, and iterate than one whose members each carry a type, which is exactly why the language keeps `enum` and `variant` apart in the first place. It earns a name for the same reason it earns a keyword.

Finding the name took a couple of wrong turns worth recording, because each was tempting. The first was **scalar**, from the cardinality framing — product multiplies, sum adds, and this one just counts. Elegant on paper, and dead on arrival at the reader: "scalar" means *a single primitive value* in nearly every language a Zane user will have met, the very opposite of a set of *n* constants, and a term that fights its own reader is one we had already refused once when we turned down "matrix" for `mould`. The second was **case**, naming each member a case — but "case" was long since spent on a `variant`'s members, where a value holds one case and `match` selects them, so lending it to `enum` too would smudge the one line the whole two-construct design exists to keep sharp. We took a genuine detour through probability, where an `enum` really does look like a sample space Ω — a finite set of mutually-exclusive, exhaustive, atomic outcomes — but "outcome" and "event" drag in randomness and measure an `enum` has nothing to do with. What every dead end had circled was the same property: the members are *atomic*, structureless, equal in rank. And the language already had a word for that, sitting in the enum chapter since it was written — its members are "uniform, interchangeable **peers**." So `peer mould` was less coined than promoted: it names the category from the `enum`'s own defining trait, collides with nothing, and imports no baggage.

The honest note is what the name gives up next to its siblings. "Product" and "sum" advertise an operation — ×, + — and "peer" does not; it describes the members, not an arithmetic. But that is the truthful trade, because the `enum`'s shape is not a third operation beside the other two, only the all-units corner of sum, and a name promising an algebra it does not extend would be the same overclaim we spent the naming on avoiding. `peer mould` says what the category *is* — a flat set of uniform peers — and leaves the arithmetic to the two shapes that actually have one.
