# Stories: Functions, Methods, and Lambdas

> **See also:** [`spec/functions.md`](../spec/functions.md) — the rules these chapters explain.

## Pulling methods out of the type body

The verb model reads, in the spec, like a top-down unification: one callable, five kinds, each selected by a marker. That is the shape it *settled* into, but it is not how it was reached. We did not set out to prove that methods, functions, constructors, operators, and lambdas were secretly one thing. Each was designed on its own terms, and the unity was something we noticed afterward and gave a name to. The honest starting point was much smaller and much more particular: a dislike of where methods live in most languages.

In the mainstream object model a method is written *inside* the type's body — a function nested in the `class` or `struct` braces, with an implicit receiver. Two things about that always grated. It **groups** the behavior with the data, which sounds tidy until you want to add a method to a type you did not declare: the body is closed, so extension becomes a second-class mechanism bolted on the side (`extension`, free functions that wish they were methods, `impl` blocks that still cluster by type). And it **indents** — every method sits a level deeper than a free function for no reason but its enclosure, and in a long body that enclosure is exactly what you lose track of. Scrolling through a hundred lines of indented functions, all sharing one unstated `this`, you forget which type you are even inside; the receiver is the most important fact about each method and it is the one fact the syntax has pushed off-screen. We could have kept methods in the body and accepted this, the way nearly every OO language does, or grouped them in `impl` blocks the way Rust does — but an `impl` block still gathers methods by type and still indents them, so it fixes the extension complaint only halfway and the indentation one not at all.

So we pulled the method out. A method became an ordinary package-scope declaration whose *first parameter* is the receiver, named `this`. Nothing encloses it; nothing indents it; and the receiver, instead of being an invisible ambient binding, is written right there at the front of the parameter list where you cannot miss what it is. Extension stops being a special feature at the same stroke: if a method is just a package-scope declaration with a `this` first parameter, then any package can write one on any type, and cross-package extension methods need no new machinery — they are the ordinary case, not an escape hatch (see [`functions.md` §6](https://github.com/zane-lang/spec/blob/54ac140005b0f4f330b24e86e0351bfd74b8fa25/spec/functions.md#6-method-name-resolution-and-extension-methods)).

That move forced a question it did not answer: if a method is just a function with a differently-named first parameter, what is *left* of the distinction? Why have methods at all, rather than declaring every operation a free function? The answer we settled on is that the `this` marker has to *earn* its existence by granting something a plain function cannot have, and the thing it grants is private-field access — the right to read and write the `_`-prefixed fields of the receiver type ([`functions.md` §2.2](https://github.com/zane-lang/spec/blob/54ac140005b0f4f330b24e86e0351bfd74b8fa25/spec/functions.md#22-this-grants-private-field-access)). Name your first parameter `this` and you are writing against the type's internals; name it anything else and you are a function that sees only the public surface. This is what makes an ordinary read-only method a meaningful category and not just a free function wearing a receiver: it has a deeper connection to the type, and the `this` token is what declares that connection. The privacy travels with the token, not with the package — a method declared three packages away still reaches `_` fields, because it is the `this` marker that grants the access, not home-package membership. It is the same orthogonality that lets `init{ }` ride on the name-is-a-type marker for constructors: one small piece of surface form, one capability unlocked, no separate mechanism.

Only once methods lived at package scope did the rest of the collapse become visible, and it became visible without being engineered. A constructor was already a package-scope declaration whose body ends in a value; giving it the marker "the name is a type" made its return type implicit and unlocked `init{ }`, and it was otherwise an ordinary declaration. An operator was the same declaration with a symbol for a name. A lambda was the same declaration with the name removed. We had not built these to match; they matched because each was, underneath, *a package-scope thing with a statement body that runs to do work.* The unified verb model in [`functions.md` §8](https://github.com/zane-lang/spec/blob/54ac140005b0f4f330b24e86e0351bfd74b8fa25/spec/functions.md#8-the-verb-model-and-capability-markers) is the recognition of that fact, written down after it was already true — one verb, and a handful of markers that each toggle one capability, rather than five parallel features with five sets of rules.

Naming the thing we had found took some care, because the good words were taken or slanted. **`callable`** and **`routine`** both lean too far toward the function side of the family — they read as synonyms for "function," which is one member of the set, not the set — and `callable` is already a term of art elsewhere. **`procedure`** carries the opposite bias: in most traditions it implies a call made *for its side effects*, with no return value, which excludes exactly the constructors and value-returning functions we most needed the umbrella to cover. Reusing **`function`** as both the umbrella and one of its members would have been the worst of the three, collapsing the very distinction the word had to help us draw. **`verb`** was unclaimed, and it says the right thing: a verb is a word for *doing something*, and the trait shared by every member of the family is precisely that its body executes to do work. The name also draws its own boundary cleanly — a subscript is *not* a verb, because its body does not run to do work; it projects a place, designating where an element lives rather than computing a value ([`functions.md` §2.9](https://github.com/zane-lang/spec/blob/54ac140005b0f4f330b24e86e0351bfd74b8fa25/spec/functions.md#29-subscripts-are-place-projections)). A thing that only *points at* storage is not doing anything, so it falls outside the word, and the word tells you why.

The cost of the marker model is that it loads meaning onto small, quiet pieces of surface form. Renaming a first parameter from `this` to `node` does not look like it changes anything, but it silently demotes a method to a function and revokes its access to `_` fields; the capability is real but invisible except through the token that grants it, so an edit that reads as cosmetic can change what the body is allowed to touch. And the decision to let privacy ride on the `this` marker regardless of package is a genuine loosening: `_`-field privacy is a convention the whole ecosystem is trusted to honor, not a wall the compiler builds around a module. Any package may declare a `this` method on your type and reach inside it. We judged that acceptable — the `_` prefix makes the reach visible at every use, and the alternative of package-sealed privacy would have taken extension methods back out of reach — but it is a real transfer of trust from the compiler to the naming convention, not a free consequence of the design.

## Mutation you can see at the call site

Pulling the receiver into the parameter list put it in plain sight at the *declaration*; the next question was what the receiver looks like at the *call*. The reasoning about *what* a method may mutate — only `this`, only through the receiver, one writable object per call — belongs to the effect model and is told in the [effects story](effects.md#where-mutation-is-allowed-to-reach); it was largely settled before this document had to say anything. What was left for the function surface to decide was narrower: whether a caller can *see*, at the call site, that a call is going to mutate the thing it is called on.

Most languages answer no, and not by oversight. In `player.reload()` there is nothing in the call to tell you whether `reload` changes the player or merely reports on it; the mutability lives in the method's declaration, or in the reference-ness of a parameter, and the call site is silent about it. We could have followed that — one call syntax, `mut` visible only where the method is declared — and it would have been less to learn. We rejected it because it hides exactly the fact we most wanted legible. If the language has already gone to the trouble of guaranteeing that a call mutates at most one object, the receiver, then a reader should be able to *collect* that guarantee by eye, without opening a single signature. So a read-only call is written with `:` and a mutating call with `!` ([`functions.md` §2.5](https://github.com/zane-lang/spec/blob/54ac140005b0f4f330b24e86e0351bfd74b8fa25/spec/functions.md#25-call-markers-are-part-of-the-surface-syntax)), and the two are not interchangeable: calling a `mut` method with `:` is illegal, and calling a read-only one with `!` is illegal. `node:scaledId(...)` cannot touch the node; `node!setScale(...)` can. You read the mutation off the punctuation.

The other road we did not take was the C++ one — make mutation a property of a *reference parameter*, pass the receiver by mutable reference, and let that carry the write permission. That buries the signal even deeper than the declaration-only approach: now mutability is a property of a type spelling somewhere in the parameter list, and the call site shows nothing at all. It also reintroduces the mutable-reference machinery the memory model was built to avoid. Putting the whole signal in a single call-site character keeps mutation visible without any of that apparatus.

The cost is a deliberate redundancy. The mutating-ness of a call is now stated twice — once by `mut` on the declaration, once by `!` at every call — and the compiler checks that the two agree. That is duplication, and duplication is usually a smell; here it is the point, because the second statement is the one the reader gets for free at the call site, and the check turns a possible mismatch into a compile error rather than a silent wrong assumption. The inherent price is that a method's mutability is now part of how it is *called*, not just how it is written: the read-only-versus-mutating distinction is welded to the call syntax, so the two capabilities can never share a spelling. Two of the surface's scarce sigils are spent buying that, permanently.

## Overloading on shapes, and only shapes

With calls marked and receivers in the parameter list, a call is fully described by the shape of its arguments — and that is exactly what we decided overload resolution is allowed to look at, and nothing else. Two declarations in the same package are the same overload if they have the same ordered parameter types ([`functions.md` §4.1](https://github.com/zane-lang/spec/blob/54ac140005b0f4f330b24e86e0351bfd74b8fa25/spec/functions.md#41-overload-identity-is-parameter-types-only)). Names do not distinguish them, because a call supplies arguments positionally and never sees a parameter's name. Return type does not distinguish them, because a bare call in a discarding position constrains nothing about what comes back, so return-type overloading would leave calls that no context could resolve. And `mut` does not distinguish them, because — as the effect model has it — the *call contract* of a read-only and a mutating method with the same parameters is structurally identical; `mut` changes what the body may do, not what shape the call has, so it cannot be the thing that tells two calls apart.

One case looks like it should be allowed and is specifically forbidden: two overloads that differ only by `&` on the same parameter position ([`functions.md` §4.3](https://github.com/zane-lang/spec/blob/54ac140005b0f4f330b24e86e0351bfd74b8fa25/spec/functions.md#43-valid-overloads-differ-by-arity-or-parameter-type)). The reason is that a `&T` parameter and a plain `T` parameter are *called with identical syntax* — the caller writes the same argument either way, and whether a new guest is created is decided by the parameter mode, not by anything at the call ([`functions.md` §2.8](https://github.com/zane-lang/spec/blob/54ac140005b0f4f330b24e86e0351bfd74b8fa25/spec/functions.md#28--and-swallowing-method-parameters)). Two candidates that a call cannot tell apart are not an overload set the compiler can resolve; rather than invent a tiebreak nobody could see, we make the declaration illegal and ask the author to rename one or pick a single signature.

That leaves the order in which candidates are tried, and here the pressure was to keep implicit conversions from ever surprising you. Resolution runs in three phases: an exact **direct** match first, then a **generic** match that infers the callee's type and number parameters from the argument types, then an **implicit** match that is allowed to insert implicit constructors at coercion sites ([`functions.md` §5](https://github.com/zane-lang/spec/blob/54ac140005b0f4f330b24e86e0351bfd74b8fa25/spec/functions.md#5-overload-resolution-with-implicit-constructors)). The ordering is the whole content of the rule. If an exact match exists, it wins outright, so adding an implicit conversion elsewhere in the program can never quietly steal a call that already had a precise home. And the generic phase runs *before* implicit conversions and without them, so type inference is never asked to discover an unknown destination type *through* a coercion — inference reads types that are actually present on the arguments, and implicit construction is a last resort that only runs once both exact and generic matching have come up empty. Implicit conversions stay a convenience at the edges rather than a force that reshapes which function you called.

The cost here is honestly small, which is worth saying plainly rather than dressing up. Restricting overload identity to parameter shape does close a couple of doors other languages leave open: you cannot write two methods that differ only in return type and let context choose, and you cannot pair a read-only and a mutating overload under one signature and one name — the mutating one is reached by `!` on a declaration whose parameters differ, or it is a different declaration. Those are real expressive narrowings, but they are narrow, and each one buys a call site that resolves the same way no matter what conversions or context surround it.

## Names that are not values

Everything so far treats a callable as something you *invoke*; the last decision is that in this language that is the *only* thing you can do with one. A method, a function, an operator — its name may appear in call position and nowhere else. You cannot write the name of a function as a value, store it, or pass it; `+` can be called but not handed around ([`functions.md` §7.1](https://github.com/zane-lang/spec/blob/54ac140005b0f4f330b24e86e0351bfd74b8fa25/spec/functions.md#71-callables-cannot-be-referenced-as-values)). This looks like a restriction until you see what it removes. An overloaded name is not one thing; it is a *set* of candidates that only collapses to a single member when arguments arrive at a call site and select one. In value position there are no arguments, so there is nothing to collapse the set — and if you then tried to pass that unresolved set into a parameter that was *itself* overloaded, both sides would have to choose at once with no information on either to choose from. Making callables call-only deletes that entire class of ambiguity by construction, which is the same reason operators are safe to overload freely ([`operators.md` §5.3](https://github.com/zane-lang/spec/blob/6d7ed465709afc61d30009c748589ef9286876ae/spec/operators.md#53-operators-are-call-only)).

But callbacks are real, and a language needs a way to pass behavior as data. The lambda is that way, and it works precisely because it is built to be the thing an overloaded name is not: a single value with one exact type. A lambda literal writes its own complete type — parameter types, return type, abort type, and `mut` — and infers nothing from its surroundings ([`functions.md` §7.2](https://github.com/zane-lang/spec/blob/54ac140005b0f4f330b24e86e0351bfd74b8fa25/spec/functions.md#72-lambdas-are-self-typed-function-values)). Because it carries its whole type, it is already collapsed to one member; there is no set to resolve, so it can be passed straight into an overloaded receiver by exact shape match, no circularity anywhere. To *name* a function value you declare a lambda-variable — a symbol bound to a lambda literal — and because a symbol has one type and can never accumulate an overload set, the name is always unambiguous in value position where a bare callable name would not be ([`functions.md` §7.3](https://github.com/zane-lang/spec/blob/54ac140005b0f4f330b24e86e0351bfd74b8fa25/spec/functions.md#73-lambda-variables-hold-function-values)). The self-typing is not ceremony; it is the property that lets a function value coexist with overloading at all.

Two consequences of the lambda's design are worth naming as costs, because they are the parts a later reader will want to have been warned about. Lambdas do not capture: every value a lambda uses must be passed as a parameter or reached through explicit storage, never closed over from the surrounding scope ([`functions.md` §7.4](https://github.com/zane-lang/spec/blob/54ac140005b0f4f330b24e86e0351bfd74b8fa25/spec/functions.md#74-lambdas-do-not-capture)). That keeps data flow explicit and keeps effect and concurrency analysis tractable — a function value cannot smuggle in hidden mutable state for a parallel task to trip over ([`concurrency.md` §5.2](https://github.com/zane-lang/spec/blob/54ac140005b0f4f330b24e86e0351bfd74b8fa25/spec/concurrency.md#52-lambdas-do-not-capture)) — but it is a real ergonomic price: closures are a convenience most languages grant, and here every dependency rides in the open. It also means there are no bound method references: because nothing is captured, there is no syntax that quietly tucks a receiver inside a function value, so code that wants a receiver later must keep it in ordinary storage and pass it back explicitly at the call. And a function value that is itself *generic* — polymorphic over its own type or number parameters — is left unspecified in this version, not because overloading cannot cope with it (a generic function type is still one unique parameter shape) but because its runtime representation, monomorphization versus dictionary passing, is a memory-model question we have not answered; that gap is owned by the [generics story's deferred account](generics.md#deferred-what-the-model-promises-but-does-not-yet-deliver) alongside the other places the model promises more than it yet delivers.
