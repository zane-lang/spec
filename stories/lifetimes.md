# Stories: Lifetimes

> **See also:** [`spec/lifetimes.md`](../spec/lifetimes.md) — the rules these chapters explain.

## Inheriting a debt: safety without a borrow checker

This story begins where the [memory story](memory.md#safety-without-a-collector-and-without-lifetimes) ends. That story turned down two things at once — a tracing collector and a lifetime vocabulary — and built the anchor so that a ref stays valid even when its value moves. But the anchor only answers *where is the value now*; it says nothing about *how long may a ref live*. A ref that resolves correctly through its anchor is still a catastrophe if it outlives the owner it tracks: the owner drains, its storage is reclaimed, and the anchor now points at a corpse. Something has to guarantee the owner outlives every ref derived from it. In Rust that something is the borrow checker, and the memory story explicitly refused it. So the debt the memory story took on — "refusing the borrow checker means refusing the thing that makes the move problem go away by fiat" — comes due here, and not only for moves: the whole question of ref safety and ownership transfer has to be settled without a single line of lifetime inference.

The tempting road was to keep just a *little* inference — track where each ref flows, compute the region it must stay valid for, and check that against its owner. That is a borrow checker wearing a smaller hat, and it loses on the same ground the memory story already chose: it makes the compiler carry a proof the source does not state, and it forbids whole patterns with errors a beginner cannot read. We wanted the rule a programmer could run *in their head* while reading a single function — the language-wide preference for a readable rule over a more powerful one, the [foundations bet](foundations.md#strictness-is-the-performance-model) applied to lifetimes. So the resolution is almost aggressively simple: every lifetime question is decided by comparing **lexical declaration scopes**, and nothing else. An `&` may be assigned only when the target's owner is declared in the same or a higher scope than the ref ([`lifetimes.md` §1.1](https://github.com/zane-lang/spec/blob/9abc6748ebcbf3b27a011f7729b78f98f110d9f8/spec/lifetimes.md#11--assignment-uses-owner-scope)); a value may move into a new owner only when that owner sits in the same or a higher scope than the source ([§1.4](https://github.com/zane-lang/spec/blob/9abc6748ebcbf3b27a011f7729b78f98f110d9f8/spec/lifetimes.md#14-destination-scope-must-contain-or-match-source-scope)). A higher scope strictly outlives a nested one — that is what lexical nesting *means* — so "the owner is declared no deeper than the ref" is exactly "the owner outlives the ref," read straight off the source with no solver in between.

The cost is the honest one for any syntactic approximation of a semantic property: it is **conservative**. There are programs that are perfectly safe — a ref whose owner happens to outlive it through some control-flow path the scopes don't reflect — that the scope rule rejects anyway, because it judges by where things are *written*, not by what actually happens at runtime. We accepted that deliberately. The whole bet of the language is that a rule you can see beats a rule that sees more; a borrow checker accepts more programs and pays for it in a proof obligation and an error vocabulary, and that is precisely the trade the memory story already declined on the program's behalf.

## What may be moved: keeping ownership subtrees whole

With scope settled, the next pressure is the move itself: of all the things you could write on the right of a transfer, which ones are the language willing to *consume*? The unprincipled answer — anything that names a value — breaks the owner-uniqueness invariant the memory model rests on the moment you allow it to reach *into* an existing owner. So the rule narrows move-sources to two forms that share one property: each names a value **owned by no one else yet** ([`lifetimes.md` §1.2](https://github.com/zane-lang/spec/blob/9abc6748ebcbf3b27a011f7729b78f98f110d9f8/spec/lifetimes.md#12-move-sources-are-owning-symbols-or-owned-verb-results)). A direct owning symbol is owned only by its binding; a verb that returns an owned `T` hands back a fresh temporary owned by nobody at all. Moving either re-parents nothing, because there was no parent to disturb.

Everything else is a projection *into* an existing owner, and that is the road not taken. The obvious wish is to move a field out — `Truck(car.engine)` — or an element out of a container — `Garage(cars[1])`. Both are rejected, and the reason is that they would tear a hole in a live ownership subtree. If `car` still exists and you have spirited its engine away, `car` now owns a gap the type system swears is full; if a container yields up element 1, every index past it shifts or the slot goes hollow, and the container's own move-and-drop machinery — the very thing the [memory story's subscript rule](memory.md#where-a-new-ref-may-come-from) was built to protect — has no way to account for the theft. We could have tried to track the holes, marking fields and slots as moved-from and forbidding their later use, but that is exactly the per-element flow bookkeeping the memory model exists to avoid. So the language forbids the move at the source instead: a field or element is not a move-source, full stop. The containing object may be moved *as a whole* when it is itself a move-source, which keeps the natural-grained operation available and only forbids the fine-grained theft.

The cost lands on anyone who genuinely wanted one piece out of a structure: they cannot have it by moving. They must either move the whole container or design the piece to be independently owned from the start. That is a real expressiveness loss, and it is the price of the guarantee that any value reachable through an owner is *still there* — that ownership subtrees are stable, not gappy.

## The declaration-block rule, and the flow analysis it refuses

The scope rules say *where* a value may move to; this chapter is about *when* — and it is where the design pays the most to stay simple. A naive language lets you move a symbol from anywhere it is visible, including inside an `if`, inside a loop, inside a nested block. The instant you allow that, you have signed up for **flow-sensitive move tracking**: the compiler must reason that after `if cond { consume(x) }`, the symbol `x` is *maybe-moved* — moved on one path, live on the other — and must forbid every later use that cannot prove which path ran. That "maybe-moved" lattice is a substantial chunk of what a borrow checker *is*, and the errors it produces ("use of partially moved value") are among the hardest a newcomer meets.

Zane refuses the whole apparatus with one blunt rule: a direct owning symbol may be used as a move-source **only in the exact lexical block where it was declared** ([`lifetimes.md` §1.3](https://github.com/zane-lang/spec/blob/9abc6748ebcbf3b27a011f7729b78f98f110d9f8/spec/lifetimes.md#13-moves-are-restricted-to-the-declaration-block)). A symbol cannot be moved from inside a nested block, so it can never be moved on *some* paths but not others — the block that could move it is straight-line code, and a move there either always happens or never does. The maybe-moved state simply cannot arise, so the analysis that would track it is not needed. Parameters get the same treatment by fiat: they are treated as declared at the top of the function body block, which is exactly the place a callee is entitled to consume them.

The road not taken is Rust's, and it is worth being clear-eyed about what it buys: flow-sensitive tracking lets you write the conditional move directly, and the compiler proves it sound. We gave that up. The cost is that **a conditional move must be restructured** — you compute the destination or the guard *first*, then perform a single unconditional move in the declaration block, rather than scattering moves across branches. For a value moved inside a loop the restriction bites hardest, and the answer is the same: hoist the decision out, move once. This is more rigid than Rust, unapologetically. The compensation is that there is no flow-sensitive move error to decipher, because there is no flow-sensitive move analysis at all — the rigidity *is* the simplification.

## Downgrade, not poison: why there is no use-after-move-read

The previous chapter removes conditional moves; this one removes the error class that, in other languages, a move leaves behind even in straight-line code. After you move a symbol, what is the symbol? The defensive answer most ownership languages give is *poison*: the binding becomes unusable, and touching it — even just to read — is a compile error. That answer exists because in those languages a moved-from slot is genuinely dangerous; its contents have been logically relocated and reading them would be reading a ghost.

Zane already paid for a better answer, in the memory story. The [anchor](memory.md#the-move-problem-and-the-anchor-that-never-moves) was built so a ref keeps pointing at a value *through* a move; a moved-from symbol is just one more thing that wants to follow the value to its new home. So instead of poisoning the symbol, the language **downgrades** it: after the move the symbol becomes an `&` value resolving through the anchor, still readable, pointing at wherever the value now lives ([`lifetimes.md` §1.6](https://github.com/zane-lang/spec/blob/9abc6748ebcbf3b27a011f7729b78f98f110d9f8/spec/lifetimes.md#16-moved-symbols-downgrade-to--values-and-are-no-longer-movable)). What it loses is only the ability to move *again* — it is now an `&`, and an `&` is not a move-source — which is precisely the double-move the poison rule was protecting against, achieved without forbidding the read. The consequence is stated flatly in the spec and it is a genuinely unusual property: **Zane has no user-visible use-after-move error class for reads.** The same holds across calls; a callee that consumes a caller's value leaves the caller a readable downgraded `&`, so callee-decides move semantics ([§1.5](https://github.com/zane-lang/spec/blob/9abc6748ebcbf3b27a011f7729b78f98f110d9f8/spec/lifetimes.md#15-callee-decides-move-semantics)) never strands the caller with a poisoned name.

What is striking about this resolution is how little it costs. The downgrade rides entirely on the anchor machinery the memory model had already built and paid for, so it adds no new runtime mechanism; it removes an error class rather than trading one burden for another. The only thing it asks of a reader is to remember that a move quietly demotes the symbol — `engine` is an owner before the move and an `&` after, so the same name denotes slightly different storage on either side of the move point — and that is a small thing to carry next to the use-after-move errors it retires. This is one of the places where the honest account is that the decision was close to free, because an earlier chapter's machinery had already covered its cost.

## Returning a ref without a lifetime to name it

The last hole is the one lifetime annotations were *invented* for. A function returns `&T`; the scope rules of the first chapter govern assignments *within* a scope, but a return value crosses out of the function entirely, into a caller scope the function cannot see. What stops `&Int bad()` from returning a ref to its own local `value`, which dies the instant the function returns? In Rust the answer is a lifetime parameter `'a` that ties the result's validity to an input's — exactly the `'a` vocabulary the memory story refused to make the source speak.

The replacement is, once more, a rule you can check by looking: a function may return an `&T` **only when that reference is rooted in one of the function's parameters**, with `this` counting as a parameter ([`lifetimes.md` §1.7](https://github.com/zane-lang/spec/blob/9abc6748ebcbf3b27a011f7729b78f98f110d9f8/spec/lifetimes.md#17-returned--values-must-be-rooted-in-a-parameter)). The reasoning is that a parameter's owner necessarily lives in the *caller's* scope — the caller had to hold a real place to pass it — so a ref rooted in a parameter is automatically rooted in something that outlives the call. A ref rooted in a local is rooted in something that dies with the call, so it is rejected. No annotation is needed because the rule does not try to express *which* caller scope the result is valid in; it only insists the result trace back to one the caller already owns, and that is enough to be safe.

This is coarser than `'a`, and that is the cost. Lifetime parameters can express relationships this rule cannot — a result valid for the shorter of two inputs, a struct that borrows for a named region — and Zane simply cannot say those things. The expressiveness lands, as the memory story warned, on the **library author**: a function that needs to hand back a reference must arrange for it to flow from a parameter, and a design that wanted to mint and return a fresh interior reference has to be rethought. What the caller gets in return is the same bargain the whole document makes — a returned `&`, once the function compiles, is known to rest on ground the caller already holds, and no one had to learn a borrow vocabulary to get there.

## Consumed or borrowed: the parameter that lives at the call site

The [downgrade chapter](#downgrade-not-poison-why-there-is-no-use-after-move-read) closed on a claim it had not fully earned. It said a callee that consumes a caller's value leaves the caller a readable `&`, never a poisoned name — but it never pinned down *when* a passed reference is actually consumed, and the spec, read closely, gave two answers that could not both be true. One section described passing by the callee's behavior: store the value and it moves, leave it alone and ownership stays with the caller. Another described a plain reference parameter as *swallowed* — frame-local to the callee, gone when the call returned. The first says the value survives in the caller's hands; the second says it dies in the callee's frame. A reader trying to answer the plainest possible question — `engine Engine(); car Car(engine)`, where the constructor takes `engine` and does nothing with it, now what? — found the document contradicting itself.

The contradiction was not cosmetic, because the downgrade rule had already spent the caller's safety on one of the two answers. If a swallowed value is frame-local and dies with the call, then the caller's downgraded `&` — the thing the previous chapter promised stays readable — points at a corpse the instant the callee drops it. The very mechanism that made "no use-after-move-read" true depended on the passed value outliving the call, and one of our two accounts quietly denied it. We had shipped a latent dangling ref and hidden it inside a wording disagreement.

The tempting repair was to keep the "swallowed, frame-local" reading and simply forbid the drop: make a callee that takes a reference *have* to re-home it, and error otherwise. That works, but we chased it and it kept forcing the value *out* of the caller. Re-home into the return, and the caller must now write `x = f(x)` to catch its own value coming back — a manual reassignment at every call, the ceremony we least wanted. Downgrade the caller no matter what, and we were back to spending the caller's ownership on every call that only wanted to *look* at the thing. Each variant treated the callee's frame as the value's home and then negotiated the value's escape. The negotiations were the smell.

The move that dissolved all of it was to deny the premise: a parameter is **not part of the callee's body scope at all**. It belongs to the [call-site scope](https://github.com/zane-lang/spec/blob/faa3ceb822c5950e2499f035bd4384b77e5a0806/spec/lifetimes.md#15-parameters-belong-to-the-call-site), one level up, and the callee is merely handed owning access to it for the duration of the call. Once the parameter lives above the body, the body draining says nothing about the value's fate — a local that received it falling out of scope does not drop it, because the value was never the local's to drop; the compiler moves it back up to where it came from. The plain question answers itself: the constructor that does nothing with `engine` hands it back, and `engine` in the caller was never in danger, because it never left the caller's scope.

With the parameter rehoused, "consume" stopped being a behavior we had to observe and became a boundary we could name. A verb consumes a parameter exactly when the value escapes into somewhere the caller still tracks: into [another parameter's owning storage, or into the return value](https://github.com/zane-lang/spec/blob/faa3ceb822c5950e2499f035bd4384b77e5a0806/spec/lifetimes.md#18-parameter-consumption-is-inferred). Those are the only two exits from a call that land the value somewhere the caller can name afterward — an argument it still holds, or a result it catches. Everything else the callee does with the value — reading it, threading it through locals, lending it to nested calls — leaves it inside the call-site scope, so the caller keeps it whole. This is the general shape, and it reaches past parameters: a scope may take an owner from a higher scope, provided the value goes back to that scope, and the two ways back are the two exits a call actually has.

We had briefly wanted to keep returns *out* of the exits, precisely to kill the `x = f(x)` reassignment. We let them back in, because forbidding them made constructors impossible to state — a constructor's whole job is to consume its arguments into the object it returns — and because the reassignment worry has a cleaner answer than a prohibition: a [non-`Void` return simply cannot be ignored](https://github.com/zane-lang/spec/blob/faa3ceb822c5950e2499f035bd4384b77e5a0806/spec/lifetimes.md#19-a-non-void-return-value-cannot-be-ignored). If the value can ride out on the return, then dropping the return on the floor would drop the value, so we require every returned value to be bound or consumed. The constructor case then needs no special pleading: `car Car(engine)` consumes `engine` into the returned object, the caller binds `car` because it must, and `engine` downgrades to an `&` that now resolves into the car — every rule already in place, nothing added for constructors.

The last question was how a *caller* knows which happened, since `island!startMatch(player)` and `boat!loadCar(car)` look identical and yet one hands `player` back while the other swallows `car` into the boat. We refused to answer it with a new annotation. Zane already infers effect levels rather than making the source spell them, and consumption is the same kind of fact, so we made it inferred too: the compiler computes, per parameter, whether a verb consumes or borrows it, starting from the verbs that store or return directly and propagating outward to their callers. What keeps this from becoming the flow-sensitive tracking we spent [an earlier chapter](#the-declaration-block-rule-and-the-flow-analysis-it-refuses) refusing is the declaration-block rule: a parameter can only be moved in straight-line code, so it goes into a local or out through an exit as two separate static cases, never consumed on one branch and borrowed on another. The classification is a single bit per parameter, and it decides the caller's state — borrowed leaves a full owner, consumed downgrades to an `&` or is caught as the return — which is the refinement the downgrade chapter was missing: it is not that every call downgrades the caller, but that only a *consuming* one does.

The costs are real and we name them. The inference is interprocedural, so a verb's parameter modes are no longer a local fact — they depend on the verbs it calls, and a change deep in a call chain can flip a caller's parameter from borrowed to consumed. We accept the whole-program coupling because the alternative was an annotation on every reference parameter, and the language's standing bet is to infer what it can rather than make the source carry it. A sharper-edged cost is that a program's well-formedness now depends on the build: a plain `T` parameter a verb only reads should be an `&T`, and we make that an error *only in release builds*, letting development builds run with the looser form so a half-written function still compiles. Two builds can therefore disagree on whether the same source is legal — a genuine wart, chosen deliberately because the strict form is the one that reaches users and the loose form is the one that helps while the code is still moving. And there is one thing this design refuses to let you express at all: you cannot take a reference and destroy it *early*, before the caller's scope ends, because a passed value always outlives the call. Destruction stays tethered to the owning scope draining, exactly as the lifetime rules demand; a function that wants a value gone cannot grant that wish, and that is the price of a parameter that always belongs to someone above.

## The signature is the whole contract: retiring inferred consumption

The previous chapter closed on a design we were briefly proud of: consumption as an inferred fact, computed per parameter the way effect levels are, so the source never had to spell out whether a passed value came back. Reading it again with colder eyes, we found that the thing we had actually built was a rule you could not run in your head — the one property the [first chapter](#inheriting-a-debt-safety-without-a-borrow-checker) swore every lifetime rule in this document would keep. Standing at a call site, looking at `something(car)`, a reader had no way to answer the plainest question there is: *is `car` still mine after this line?* The answer lived somewhere else entirely — inside `something`'s body, and inside the bodies of everything `something` called, since the classification was interprocedural. The signature, the one thing right there on the page, said nothing. Whether `car` survived as an owner or quietly became a tether was a fact you had to go read another function to learn, and it could flip out from under you when that other function changed. We had traded an annotation we didn't want to write for a magic we couldn't see, and on this codebase that is a bad trade every time.

The tempting patch was to keep the inference and merely surface it — show the computed consumed-or-borrowed bit in tooling, let a reader hover a call and be told. But that concedes the whole game. If you need a tool to tell you what a line of your own code does to your own value, the language has already failed the read-it-in-your-head test, and dressing the failure in a hover tip does not undo it. The other patch was to keep inference for the common case and add an annotation for when you wanted certainty, which is the worst of the three: a rule that is sometimes inferred and sometimes written, so a reader must first work out *which mode is in force* before working out anything else. Every repair kept the callee's body in charge of the caller's fate and then tried to make the body's verdict easier to fetch. The verdict itself was the problem.

So we cut it. Passing an owning value to a plain `T` parameter [downgrades the caller's symbol to a tether](https://github.com/zane-lang/spec/blob/b932c539f2cb9c313fa1efabcd482481ec3b107a/spec/lifetimes.md#18-passing-an-owner-to-a-t-parameter-downgrades-it-to-a-tether), full stop, and the callee's body gets no vote. `T` in the signature means *the caller gives this up*; `&T` means *the caller lends it and stays the owner*. That is the entire contract, and it is legible from the call site because it is legible from the *signature* — the only thing a caller was ever reading. A verb that just glances at its argument and a verb that squirrels it into a field now look identical to the caller, and that is exactly right: both asked for `T`, both took an owner, and reading the value instead of storing it does not un-ask the question the signature posed. As the decision was put to us: nothing in the body changes what the signature already said. If a verb only wants to look, it must say so in the one place that speaks to the caller — by taking `&T`.

The mechanism this rests on was already built and already paid for. A [passed value still belongs to the call-site scope](https://github.com/zane-lang/spec/blob/b932c539f2cb9c313fa1efabcd482481ec3b107a/spec/lifetimes.md#15-parameters-belong-to-the-call-site), so it outlives the call whatever the body does with it; and the [downgrade-not-poison](#downgrade-not-poison-why-there-is-no-use-after-move-read) rule already turned a moved-from symbol into a readable tether through the anchor rather than a poisoned name. Passing to a `T` parameter is now simply a move like any other — the argument is a move-source, the symbol downgrades, the anchor keeps it resolving to wherever the value comes to rest. The elaborate part of the old design had been the *exception*: the borrowed case, where we chose *not* to downgrade because the body happened not to store the value. Deleting the exception removed no machinery; it removed a special case, and left passing-an-owner as an ordinary instance of the move rule this document already had. The old wart went with it — the *passing mode* no longer depends on the build, because there is no read-only-`T`-should-be-`&T` judgment left to relax in development and enforce in release. A read-only `T` is simply legal, and simply downgrades the caller, like every other `T`. Doing *nothing* with a parameter is still a release-build error, but that is the general unused-parameter rule, owner or not, and never had anything to do with consumption.

Getting an owner back, when you genuinely need it, is an explicit act rather than an inferred gift: the verb returns the owner and the caller binds it. That gives clean names to the two ways a verb can swallow an owner you hand it, told apart by its return — it **relays** it (`T`, and returns an owner you can rebind) or **consumes** it (`T`, returns nothing, keeps the value) — beside the `&T` tether, which swallows nothing and leaves you the owner outright. `car2 Car = repair(car)` says in plain sight that `repair` relayed the car and handed it back; there is no inferred alternative hiding behind it in which the same line would have left `car` an untouched owner. The reassignment is not noise obscuring a simpler truth — it *is* the truth, written where you can see it.

We first drew that last line too hard. To stop a dropped return from stranding the caller's tether, we made binding *mandatory* — a non-`Void` return could not be ignored. But the danger dissolves the moment you notice the value need not die when it goes unbound: it can [float to the enclosing scope](https://github.com/zane-lang/spec/blob/b932c539f2cb9c313fa1efabcd482481ec3b107a/spec/lifetimes.md#19-an-ignored-owned-result-floats-to-the-enclosing-scope), an anonymous owner living exactly as long as any scope-owned value, the caller's tether resolving into it. Nothing dangles, nothing is silently destroyed. And floating turned out *more* uniform than the mandate it replaced: passing an owner already leaves you a tether to a value that floats at the call-site scope, and ignoring a returned owner now does the identical thing — one rule, not a rule plus an exception. So binding stopped being an obligation and became a choice: take owning privilege, or don't. The choice even reads at the call site, since a caught return is exactly the mark that the caller wanted ownership back.

The cost is real and we name it. The convenient case — hand a value to a verb that only reads it and keep using it as an owner afterward, no ceremony — is gone; you now pass `&T` for that, deciding at the parameter's declaration that reading was all you meant. The loss is a genuine one in terseness, paid at every call that wants to lend-and-keep: what the inference used to hand back for free now costs either an `&T` at the parameter's declaration or a bound return at the call site. We took it deliberately, because the terseness it bought was precisely the terseness that hid the answer to *is this still mine*, and the standing bet of this whole document is that a rule you can see beats a rule that saves a keystroke. We also gave up the small elegance of parameter modes being inferable rather than declared: `&` on a read-only parameter is load-bearing now where it used to be optional, and a signature that gets it wrong is wrong at the call site rather than merely suboptimal. That, too, is a fair price for a contract that says what it means in the one place everyone reads.
