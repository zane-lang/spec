# Stories: Concurrency

> **See also:** [`spec/concurrency.md`](../spec/concurrency.md) — the rules these chapters explain.

## Parallelism you can't see, concurrency you must ask for

Two unrelated things travel under the one word *concurrency*, and most of the trouble in this part of the design came from refusing to let them stay merged. One is going faster: running work that does not depend on other work at the same time, so a multi-core machine finishes sooner. The other is changing what the program *computes*: letting independent lines of execution interleave, race, and produce results that depend on who got there first. The first is an optimization the programmer should never have to think about; the second is a semantic decision the programmer must make deliberately, because it can change the answer. A language that hands the programmer one knob labelled "concurrency" forces them to reason about meaning every time they only wanted speed, and to reason about speed every time they meant to change meaning.

So the first move is to split the word in two and give each half opposite rules. **Parallelism** is the compiler's business and is defined to be *unobservable*: the compiler may run provably independent work at once, but only when doing so cannot change the output — it may move timing, never results ([`concurrency.md` §2.2](https://github.com/zane-lang/spec/blob/d8fda9f8695651bf68ac27f986bc7b550721641a/spec/concurrency.md#22-parallelization-of-the-residual-graph)). **Concurrency** is the programmer's business and is the opposite: `spawn` explicitly opts *out* of serial equivalence, and program results may legitimately depend on scheduling ([§3.7](https://github.com/zane-lang/spec/blob/d8fda9f8695651bf68ac27f986bc7b550721641a/spec/concurrency.md#37-no-serial-equivalence-guarantee)). The same machine runs both, but they answer to different masters: one may never be noticed, the other must always be asked for.

The unobservable half only works because an earlier document already paid for it. The compiler can parallelize the residual graph only where it can *prove* two pieces of work do not conflict, and that proof is exactly what the [effects story](effects.md#inferring-effects-instead-of-naming-them) built — effects are inferred and visible in the structure, so independence is a property the compiler can read rather than guess. The same machinery decides what disappears before the runtime graph even exists: a *Total Pure* call with known inputs is evaluated at compile time and removed entirely ([§2.3](https://github.com/zane-lang/spec/blob/d8fda9f8695651bf68ac27f986bc7b550721641a/spec/concurrency.md#23-total-pure-and-pure-are-distinct)), which is why the [four-level ladder](effects.md#four-levels-and-the-line-between-terminating-and-merely-pure) bothered to separate *Total Pure* from merely *Pure* — that line is the line between work the compiler can erase and work it can only parallelize.

The hardest road not taken here was `async`/`await`. It is the dominant answer — Rust, C#, JavaScript, Python all encode "this might run concurrently" into the function's *type*, so a function becomes a different kind of thing the moment some caller wants to run it off the main line, and that color spreads up the call chain until half the program is dyed. Zane refuses it outright ([§5.3](https://github.com/zane-lang/spec/blob/d8fda9f8695651bf68ac27f986bc7b550721641a/spec/concurrency.md#53-no-asyncawait-syntax)): whether work runs concurrently is a property of the *call site*, chosen with `spawn`, not a permanent property of the definition. The same function is called normally in one place and spawned in another, and its signature does not change. This keeps the language out of the function-coloring trap, but it is not free: because parallelism is the compiler's call, governed by profitability heuristics over instruction counts and IO ([§2.2](https://github.com/zane-lang/spec/blob/d8fda9f8695651bf68ac27f986bc7b550721641a/spec/concurrency.md#22-parallelization-of-the-residual-graph)), the programmer cannot *force* a pure computation to run in parallel or *forbid* it — they are promised only that the answer will not change, never a particular timing. The one coarse lever they do get is `@threads` ([§2.4](https://github.com/zane-lang/spec/blob/d8fda9f8695651bf68ac27f986bc7b550721641a/spec/concurrency.md#24-thread-configuration)), a whole-program pool size, not a per-call placement. That is the deliberate shape of the bargain: total control over *meaning* through `spawn`, almost none over *timing*, and we think that is the right place to draw the line.

## `spawn`, and why it marks only a call

If concurrency must be asked for, the asking needs a syntactic home, and the shape of that home decided more than it first appears. The question was deceptively small — what, exactly, are you allowed to `spawn`? The expressive answer is "anything": a block, a branch, a loop body, any chunk of work you want to peel off. We took the restrictive answer instead. `spawn` may target a **function call and nothing else** ([`concurrency.md` §3.1](https://github.com/zane-lang/spec/blob/d8fda9f8695651bf68ac27f986bc7b550721641a/spec/concurrency.md#31-spawn-targets-function-calls-only)) — `spawn [ ... ]` and `spawn if ... ` are illegal — and the reason reaches forward into the safety rules. Every check that makes concurrency safe in Zane is read off a callee's *signature*: whether it needs `mut` access, which external resources it reads or writes. A function call presents that signature; an arbitrary block does not. Letting you spawn a block would force the compiler to derive an effect signature for a lump of inline code reaching into whatever surrounds it, which is exactly the open-ended analysis the call-only rule sidesteps. The restriction is what keeps conflict detection signature-based and statically decidable.

That wording later exposed a terminology mistake rather than a design boundary. We had used “function call” loosely to mean a call with a verb signature, while Zane uses *function* narrowly for a verb without `this`. A method call presents the same signature — including its receiver and `mut` permission — and was always intended to be spawnable for the same reason a function call is. The standing boundary is therefore **function or method calls, and nothing else**. Blocks and control flow remain outside it because they provide no verb signature for the analysis to read.

With the unit settled, the next question is how a spawned result comes back. A spawned call that returns a value binds to a symbol like any other, and the trick is what reading that symbol does: it **blocks until the call finishes** ([§3.2](https://github.com/zane-lang/spec/blob/d8fda9f8695651bf68ac27f986bc7b550721641a/spec/concurrency.md#32-spawned-values-block-on-read)). There is no future object, no `.await`, no `.get()` handle. The road not taken was the explicit future — a `Future<T>` you thread around and resolve later — and it loses on the same ground as `async` coloring: a future is a second type wrapped around the real one, and it spreads the same way. Making the bound symbol *be* the value, and putting the only blocking point at the moment of use, keeps the data dependency visible exactly where it matters — you can see, at the read, that this is where the program waits — without minting a new type to carry it.

The same instinct settles the abortable case. A spawned call that can fail still attaches its `?` or `??` handler **directly to the `spawn` expression** ([§3.3](https://github.com/zane-lang/spec/blob/d8fda9f8695651bf68ac27f986bc7b550721641a/spec/concurrency.md#33-abortable-spawned-calls-are-handled-at-the-spawn-site)), not at some later result-read site. We considered deferring the handling to where the value is read — it would mirror the blocking model — but it would have split error handling into two dialects, one for ordinary calls and one for spawned ones, and the whole point of the [error-handling model](../spec/error-handling.md) is that an abortable call is handled at the call. A `spawn` *is* the call; the handler belongs there, and the bound symbol simply carries the already-handled type. So concurrency does not get its own error story; it inherits the existing one intact.

Two smaller rules round out the call model and both follow from taking ordering seriously. The compiler never reorders `spawn`s — source order is start order, and each blocking read happens exactly where written ([§3.6](https://github.com/zane-lang/spec/blob/d8fda9f8695651bf68ac27f986bc7b550721641a/spec/concurrency.md#36-ordering-is-explicit)) — because once the programmer has opted into observable concurrency, the source is the only record of intent the compiler is allowed to honor. And a stalling call, when spawned, is *parked* rather than left to occupy a worker thread while it waits ([§3.4](https://github.com/zane-lang/spec/blob/d8fda9f8695651bf68ac27f986bc7b550721641a/spec/concurrency.md#34-stalling-functions-are-parked)), so a bounded pool can hold many more waiting calls than it has threads. The cost of the call-only discipline is paid in ergonomics: to run a compound computation concurrently you must first give it a name as a function, since you cannot wrap a block — and the no-capture rule of a later chapter sharpens that cost, because the function cannot quietly reach for its surroundings either. We judged the verbosity a fair price for a `spawn` whose safety the compiler can settle by looking only at a signature.

## The water tower: lifetimes that survive the spawn

Spawning a call creates a problem the rest of the memory model never had to face: the work outlives the statement that launched it. Up to now, lifetimes were a lexical matter — the [lifetimes story](lifetimes.md#inheriting-a-debt-safety-without-a-borrow-checker) settled every question of "does the owner outlive the ref" by comparing declaration scopes, and the [memory story](memory.md#safety-without-a-collector-and-without-lifetimes) guaranteed an object stays put for as long as its scope holds it. But a spawned call can still be running when its enclosing scope reaches the end of its lexical block. The scope's owned objects are scheduled for destruction at that brace; the spawned work may still be reading them. The lexical guarantee, which assumed execution leaves a scope exactly once and only after everything inside it has finished, suddenly has a hole the width of every outstanding `spawn`.

The obvious escapes were all ones the language had already refused elsewhere. A garbage collector would paper over the problem by keeping objects alive until nothing references them — but the whole project is built on having neither a collector nor a lifetime vocabulary, and reintroducing one just for concurrency would be a quiet surrender of that bet. Reference counting every object that might be shared with spawned work pays a runtime tax on the common case to fix a rare one. Forcing spawned calls to take ownership, or to work on copies, would make the model safe by making it useless for the ordinary thing — handing some live state to a background task. And the route most languages take — explicit join handles the programmer must remember to await, `thread::scope` and its kin — puts the safety back in human hands, where a forgotten join is a use-after-free.

The resolution keeps the lexical spirit and pushes it through the spawn: a scope does not *complete* until every `spawn`ed call inside it has completed, and its owned objects are destroyed only when the scope finally **drains** ([`concurrency.md` §4.1](https://github.com/zane-lang/spec/blob/d8fda9f8695651bf68ac27f986bc7b550721641a/spec/concurrency.md#41-water-tower-lifetime-extension)). The spec's image is a vertical water tower with a horizontal plate held in place by each still-running spawned call; the water — destruction — cannot fall past a plate that is still there. Each finished call removes its plate and the level drops to the next; only when the last plate is gone does the water reach the bottom and ordinary destruction run. What makes this satisfying is that it adds no new vocabulary and no new runtime bookkeeping the model did not already have: it is the same scope-drain mechanism the destruction rules already used, with its trigger moved from "the block ends" to "the block ends *and* all its spawns have returned." The lexical lifetime guarantee now simply extends to cover the concurrent work a scope launched, exactly as if that work were still inside the braces — which, in lifetime terms, it is.

The cost is real and worth stating plainly, because it is the kind a reader will hit. The granularity of the extension is the *scope*, not the individual object: one long-running `spawn` holds the whole scope's owned objects alive until it returns, even objects it never touched, because the drain is all-or-nothing. There is no early release. And since the language offers no cancellation — the subject of a later chapter — a `spawn` that never returns holds its scope open forever, with no language-level way to pry it loose. We accepted this as the honest shape of "no GC, no join handles": the safety is automatic and unforgettable, and the price is that liveness is coupled to the coarse unit of the scope and to the spawned call's own willingness to finish.

## Safety the compiler proves from signatures, not locks

The water tower guarantees a spawned call's objects are still *alive*; it says nothing about two spawned calls touching the same live object at once. Liveness is not enough — the classic hazard is the data race, two concurrent accessors of one object where at least one writes. The mainstream answers all push the work to runtime: locks and mutexes the programmer places by hand and hopes are correct, atomics for the brave, or a race detector that finds the bug only on the run where it happens to manifest. Each makes correctness a runtime property the type system never sees, which is precisely the kind of unprovable obligation the rest of the language works to avoid.

Zane rules the race out statically instead, and again it spends a debt the [effects story](effects.md#where-mutation-is-allowed-to-reach) already funded. The **single-writer rule** says at most one concurrent `mut` accessor may exist for any object; two spawned calls that both need `mut` access to the same object are a compile-time error, not a runtime lock ([`concurrency.md` §4.2](https://github.com/zane-lang/spec/blob/d8fda9f8695651bf68ac27f986bc7b550721641a/spec/concurrency.md#42-single-writer-rule-for-object-mutation)). For state outside the object graph — files, sockets, the world — the effect system classifies each access as read or write and serializes anything that conflicts: read/read runs free, but read/write and write/write are forced into order, and the compiler arranges it from the effect signatures without the programmer writing a single lock ([§4.3](https://github.com/zane-lang/spec/blob/d8fda9f8695651bf68ac27f986bc7b550721641a/spec/concurrency.md#43-effect-conflicts-on-external-resources)). This is the deep reason `spawn` targets only calls (from [the previous chapter](#spawn-and-why-it-marks-only-a-call)): both checks are read straight off the callee's signature — its `mut`-ness, its resource effects — and a call is the thing that has a signature to read. The safety rule and the spawn-shape rule are the same decision seen from two sides.

This is also where the prohibition on capture earns its place. Lambdas and value-blocks **must not** capture outer variables; every dependency is passed explicitly ([§5.2](https://github.com/zane-lang/spec/blob/d8fda9f8695651bf68ac27f986bc7b550721641a/spec/concurrency.md#52-lambdas-do-not-capture)). The rule looks like an ergonomic tax until you see it from the analysis's side: a capture is an access that the signature does *not* show, a side door through which a spawned closure could reach a shared object the single-writer check never counted. The alternative — allowing capture and inferring the captured effects through to the closure's call sites — is exactly the whole-closure effect analysis the no-capture rule retires, and it would make the conflict check depend on flow the signature can no longer summarize. Forbidding capture keeps the signature the complete truth about what a spawned call touches. A last sharp edge is closed by [§4.4](https://github.com/zane-lang/spec/blob/d8fda9f8695651bf68ac27f986bc7b550721641a/spec/concurrency.md#44-refs-passed-to-spawned-work-remain-independent): an `&` handed to spawned work resolves to its own value against the same owner, so rebinding the caller's `&` symbol afterward retargets only the caller's storage and cannot reach in and redirect the reference the spawned work already holds — the shared-mutable-binding hazard, removed by making the binding unshared.

The costs here are the most contentious in the document, and pretending otherwise would be dishonest. The single-writer rule is conservative at the granularity of the whole object: two spawned calls that write *disjoint* parts of one object, or that the programmer has coordinated by some scheme the compiler cannot see, are rejected anyway, because the check knows only "needs `mut` to this object," not "needs `mut` to this corner of it." Patterns that are genuinely race-free are turned away for want of a proof the signature can carry. And no-capture makes concurrent code verbose in the ordinary case — every dependency a spawned call needs becomes an explicit parameter, with nothing allowed to ride along implicitly. We took both costs deliberately: they are the price of a concurrency model with *no locks to misplace and no race detector to run*, where a program that compiles is already free of the data races the rules describe.

## What the core deliberately leaves out

Having built `spawn`, the water tower, and signature-based safety, the strongest pull is to keep going — to furnish the concurrency core with the apparatus every mature concurrent language seems to grow: cancellation and kill groups, channels, an actor or process primitive, structured shutdown. We stopped instead, and the omissions are a design position, not an unfinished to-do list. There is no cancellation, no shutdown ordering: a spawned function runs until it returns or the process is killed from outside ([`concurrency.md` §5.1](https://github.com/zane-lang/spec/blob/d8fda9f8695651bf68ac27f986bc7b550721641a/spec/concurrency.md#51-no-cancellation-or-shutdown-ordering)). And there is no language-level `Process` type, actor, or channel; long-running concurrent work is just ordinary spawned calls plus state that flows under the ownership and effect rules already in force ([§5.4](https://github.com/zane-lang/spec/blob/d8fda9f8695651bf68ac27f986bc7b550721641a/spec/concurrency.md#54-no-language-level-process-or-channel-abstraction)).

The roads not taken are well-trodden and genuinely good — structured cancellation as in Kotlin's coroutines or Go's `context`, CSP channels as in Go, the actor model as in Erlang — so the reasoning for declining each matters. Cancellation is the one we refused on principle rather than budget: a cancellable task must be stoppable at defined points, which reintroduces hidden suspension and a form of the control-flow coloring the [first chapter](#parallelism-you-cant-see-concurrency-you-must-ask-for) spent its effort removing, and it tangles with the water tower besides — what does it mean to cancel work that a draining scope is waiting on? Channels and actors we declined on a different ground: they are orchestration *philosophies*, and baking one into the core would commit every Zane program to it. They belong in libraries, built from spawned calls and owned state, where a program can choose the model that fits rather than inherit ours. The throughline is the language-wide preference for a small, predictable core over a feature-rich one.

This is the chapter where the costs are heaviest, and they compound the ones before it. No cancellation means runaway or no-longer-wanted work cannot be stopped from within the language; joined with the [water tower](#the-water-tower-lifetimes-that-survive-the-spawn), a `spawn` that hangs holds its scope's objects alive indefinitely with no recourse short of killing the process. The orchestration patterns working programmers reach for daily — timeouts, fan-out/fan-in with early cancellation, graceful shutdown that drains in order — must be assembled by hand or supplied by libraries, and a few of them, true preemptive cancellation chief among them, simply *cannot* be built safely on top of this core at all, because the core gives nothing to build them from. That is the sharpest admission in the concurrency design: the model is deliberately small enough that some things everyone expects are not merely absent but out of reach, and we are betting that a concurrency core you can hold entirely in your head, with no hidden control flow, is worth more than one that ships every convenience and the unpredictability that comes with them.

## Value-typed mutation: closing the aliased-write gap

The safety rule the [previous chapter](#safety-the-compiler-proves-from-signatures-not-locks) was proud of turned out to have a hole, and the hole is worth the whole story of how the rule was rebuilt. The single-writer rule read the *receiver instance* off each spawned `mut` call and forbade two of them from naming the same object. What it could not see was mutation reaching an object by a *second path*. Zane lets you mutate through an `&`, and a reference type could hold an `&` field, so a wrapper holding `&Engine` and a bare `Engine` present two different receiver types at their signatures while both driving writes into the one `Engine` — one directly, one through the wrapper's alias. The per-instance check passed; the data race did not. This was filed as issue #98, deliberately left open, with the spec and the chapter above left standing as the honest record of a rule that did not yet hold.

Closing it looked, at first, like a choice between two bad options. One was to make the single-writer check *see* the second path — compute each spawned call's transitive mutation set and prove that two of them cannot alias. That is whole-program alias analysis, and it is exactly the non-local reasoning the rest of the language spent its effort avoiding; the [lifetimes design](lifetimes.md#inheriting-a-debt-safety-without-a-borrow-checker) bought its safety by *looking* at a declaration, not by tracing a program, and we were not willing to give that up here. The other was to serialize conservatively — force any two `mut` calls whose reachable types could overlap into sequence — which is sound but throws away the parallelism that was the point. Neither was a design we wanted to sign.

The way out was not to *restrict* mutation but to *enable* it, and that is the part worth getting right, because the restriction everyone remembers is a corollary of it rather than the idea itself. A value type's alias-freedom was exactly the property that would make concurrent mutation safe — but it was useless while value types were immutable, because there was no way to express shared *mutable* state as a value in the first place. So the move was to make value types **mutable in place**. That is less small than it sounds: mutating a *copy* helps no one, so a mutation has to reach the caller's real storage — a borrow, not a copy — and a mutable-yet-alias-free record did not exist as a kind at all. Supplying one is what forced `struct` and `class` apart into a single value/reference axis — a type is a value unless marked `#` ([`foundations.md` §7](https://github.com/zane-lang/spec/blob/9ab62085ed13f22b0a4897425852327264bd022d/spec/foundations.md#7-identity-is-opt-in)) — with the value side transitively alias-free by construction, holding no `&` and no reference field anywhere downstream ([`memory.md` §2.10](https://github.com/zane-lang/spec/blob/9ab62085ed13f22b0a4897425852327264bd022d/spec/memory.md#210-value-downstream-enforcement-transitive-value-only-field-restriction)). Only *then*, with a mutable alias-free value in hand, does the concurrency rule fall out for free: **a spawned call may mutate only a value-typed receiver** ([`concurrency.md` §4.2](https://github.com/zane-lang/spec/blob/9ab62085ed13f22b0a4897425852327264bd022d/spec/concurrency.md#42-concurrent-mutation-requires-a-value-typed-receiver)). Because a value's reachable graph contains no reference, there *is* no second path — the #98 wrapper cannot be built, since a type holding an `&` field is not a value type. The compiler rules out the aliased race from the receiver's type alone, which is precisely the "check by looking" the alias-analysis road would have abandoned.

Two consequences fall out that make the rule feel less like a patch and more like the shape the design wanted. First, since only value types are ever mutated by spawned work, reference types are *frozen* for the duration of the concurrency, so every concurrent read of the reference-typed object graph is safe with no further rule. Second, the old per-object single-writer check survives in a cleaner form: at most one live spawn may hold a *mutable borrow* of a given location ([§4.3](https://github.com/zane-lang/spec/blob/9ab62085ed13f22b0a4897425852327264bd022d/spec/concurrency.md#43-single-writer-per-storage-location)), and because value types carry no `&`, a location's identity is unambiguous, so that disjointness is again a spawn-site check rather than a program trace. The one thing that had to be added to keep real-time work honest is that a reader of a value another spawn is writing takes a *coherent snapshot* rather than blocking ([§4.4](https://github.com/zane-lang/spec/blob/9ab62085ed13f22b0a4897425852327264bd022d/spec/concurrency.md#44-reads-take-a-coherent-snapshot)) — copying a value *is* the snapshot — which replaces lock-based serialization for in-memory state and is why an audio reader never waits on a parameter writer.

The cost is real and it is the mirror image of the gain. Shared *mutable* state that crosses a spawn must now be expressible as a value — a self-contained tree with no internal sharing — because that is the only thing the rule lets you mutate concurrently. Genuinely graph-shaped mutable state, with aliased sub-objects or cycles, cannot be handed to two spawns at all; it has to be restructured into values or kept single-threaded. We think that is the right price, and not only because it is safe by construction: the patterns that most want concurrent mutation — a parameter block, an audio buffer, a snapshot of world state — are already value-shaped, and the ones that are not are exactly the ones where a silent data race would have been hardest to find. The rule that began as a hole in a per-instance check ended as a single sentence about the receiver's type, and the reason it can be that small is that the value/reference axis had already done the work.
