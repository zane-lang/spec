# Stories: Dependency Management

> **See also:** [`spec/dependencies.md`](../spec/dependencies.md) — the rules this story explains.

## URL identity and the two-file manifest

The first fork in any dependency system is what a package *is*. The mainstream answer is a central registry: a package is a short global name — `lodash`, `serde`, `requests` — that the registry maps to code. We could have built that, and it buys real things: memorable names, a single place to search, a curated namespace. We turned it down anyway, because a registry is a name authority, and a name authority is a single point of control, failure, and dispute. Someone has to own the mapping from `math` to a repository, adjudicate who gets the name `http`, keep the index alive for the life of the language, and be trusted not to repoint a name at hostile code. Every registry ecosystem eventually fights typosquatting, name-squatting, and the day the index goes down. None of that is essential to *fetching code*; it is essential only to the short-name convenience the registry exists to sell.

So Zane takes the URL the code already lives at as the identity. `github.com/zane-lang/math` is the package — there is nothing to register, no authority to trust, and the name is globally unique by construction because URLs already are. The thing you fetch from and the thing you name it by are the same thing, which removes a whole layer of indirection and the failure modes that come with it.

The price is paid in two coins, and they are real. First, there are no short names: source would be unreadable if every call site spelled out a URL, so the manifest assigns each dependency a local **key** — `math` — used in `import math` and as the join column between the two manifest files. That key is deliberately *only* a nickname, scoped to one project; it carries no identity and two projects may nickname the same URL differently. Second, identity is tied to *location*, so a repository that moves — an org rename, a host migration — looks like a different package, and consumers must update the URL. A registry would have absorbed that move behind a stable name. We judge the move-churn cheaper to live with than a permanent name authority to maintain, but it is a genuine thing the registry road would have given us and this one does not. That the key is a per-project nickname with no global meaning is exactly why the [remapping opt-in](#opt-in-remapping), much later in this story, is written in URLs rather than keys.

Given URL identity, a project still has to record two quite different things: what the human *asked for*, and what that request actually *resolved to*. The lazy design folds them into one file — a single manifest holding key, URL, version, and commit hash all together. We split them into `zane.coda` (intent) and `zane-versions.coda` (resolution) instead, and the split is not bookkeeping fussiness; it tracks a difference in audience and lifecycle. Intent is small, human-authored, and reviewed: "I want `math` at `v6.2.9`." Resolution is large, machine-generated, and verified: "that tag is URL `…/math` at commit `a3f8c2d`." Mixing them means every intent review — the thing a human actually reads in a pull request — is buried under churning commit hashes the human neither wrote nor should diff. Keeping intent in its own file means a dependency-bump review shows a one-line tag change, while the resolution file acts as a lockfile: a verifiable record that this tag *was* this commit, so a later moved tag is detectable, a thread [the tag-and-commit pinning](#a-tag-for-humans-a-commit-for-the-machine) picks up below. This is the same intent-versus-lock division that `package.json`/`package-lock.json` and `Cargo.toml`/`Cargo.lock` arrived at; we are not original here, and the convergence is a good sign.

The cost of two files is that they can drift out of sync — a key in one and not the other, a hand-edit to one alone. We pay it down not by merging them back but by making the toolchain refuse to guess: build-flow step 1 validates that the keys in the two files correspond exactly, and aborts on any missing, extra, or mismatched key rather than papering over it. Drift becomes a loud error instead of a silent mis-resolution, which is the trade we want — the split's danger is contained by strictness rather than by giving up the split.

## Shipping compiled objects, and rewriting their symbols

With identity and the two-file record settled, the next question is how a dependency actually arrives as something linkable. Three roads. Distribute source and compile it on the consumer's machine, the way Cargo does. Distribute prebuilt binaries as release assets attached out-of-band to a tag, the way GitHub Releases and most C package managers do. Or commit the prebuilt object files into the repository tree itself, beside the source. Zane takes the third, and it is the most unusual choice in the document, so it is worth being honest about why and about what it costs. Compile-on-install was rejected because it makes every consumer pay the full build cost of every transitive dependency, and makes "fetching a package" depend on a working local toolchain for that package's every quirk — the thing that makes a clean dependency install slow and fragile elsewhere. Release-asset binaries were rejected for a subtler reason: they make fetching depend on a host-specific API. The binary for tag `v1.0.1` lives not in git but in GitHub's (or GitLab's, or some artifact server's) release endpoint, so the fetcher has to speak each host's asset protocol, and the source and its binary no longer share one immutable identity — the tag can point at one commit while its attached assets are swapped underneath. Putting the objects *in the tree under* `build/` means a fetch is just a `git clone`, host-agnostic by construction, and the source and its compiled form share the single immutable identity of one commit. That last property is what makes the [commit-pin tamper check](#a-tag-for-humans-a-commit-for-the-machine), described shortly, cover the binaries too, not just the source.

The costs are not small and we do not pretend otherwise. Repositories carry compiled objects for every supported target triple, so they are larger and their history heavier. A package is only usable on a target whose prebuilt artifact the author actually shipped — there is no compile-from-source fallback in the normal path, so a missing triple is a hard "unsupported here." And consuming a prebuilt object means trusting the author's binary, which is a heavier trust than trusting source you could read. We mitigate the last one with an explicit escape hatch — `--from-source` compiles from the verified source checkout for anyone who does not trust the shipped object — but we keep that as opt-in rather than default precisely because the host-agnostic, identity-sharing prebuilt path is the property we are optimizing for, and making source-compilation the default would forfeit it for everyone to serve the cautious minority.

Shipping prebuilt objects raises a problem the moment two parts of one program need two different versions of the same library — your direct dependency on `math v6` and some transitive dependency's pin to `math v5`. A linker cannot hold two symbols both named `math$vec`; one definition wins and the other's callers are silently mislinked. The classic answers are all unhappy: force the whole graph onto one version (a solver, with the diamond-dependency conflicts that make `npm` and `pip` painful), or make authors bake a version number into every public name by hand (`math_v6_vec`), which pushes versioning into source and makes every API rename a version bump. Zane's answer is to make the *toolchain* do the renaming, mechanically, at fetch time. A library is compiled with its own exports carrying a placeholder marker — `!math$vec` — and when `zane add` pulls a specific version, it rewrites that placeholder into the resolved version tag, producing `v1.0.1%math$vec`. Two versions now have genuinely different symbol names and coexist in one link with no collision and no solver. The author never writes a version into their source; the `!` placeholder is a promise that the toolchain will stamp the real version in later. This is why side-by-side coexistence, not version unification, is the default behavior — the mechanism makes coexistence the cheap path.

The interesting scar on this story is the separator character, because it is where an abstract design met the ugly reality of platform linkers. The natural first pick between the version and the key was `@` — `v1.0.1@math$vec` reads cleanly and `@` is conventional for "at version." It is also actively dangerous: on ELF, `@` is *reserved* for symbol versioning (`symbol@VERSION`), so a linker reading `v1.0.1@math$vec` may decide `v1.0.1` is the symbol and `math$vec` is its version — exactly backwards — and Mach-O and PE toolchains variously reject or decorate `@` too. We went looking for a character with no reserved linker meaning on any of the three major object formats and settled on `%`. The competing candidate was `__`, which is maximally portable but reads as part of an identifier rather than as a structural delimiter; `%` was chosen over it because the symbol scheme *already* spends a non-alphanumeric delimiter — `$`, between the package key and the member — so the toolchain has already committed to carrying assembler-special punctuation through symbol names, and `%` is the same class of character as the `$` we have lived with all along, not a new kind of risk. To make the first `%` an unambiguous delimiter, `%` is also forbidden in version tags by the same path-safety validation that screens cache paths (§7), so the version-to-key boundary can never be faked by a `%` inside a tag. The lesson worth recording: a symbol-mangling scheme is not free to use whatever punctuation reads nicely, because the punctuation namespace is already colonized by linkers, and the only safe move is to pick from what they all leave alone and then reserve it.

The cost of the whole approach is a rewrite step with state: there is now a distinction between the author's original `!`-prefixed objects (which arrive in the repo's own `build/`, cached at `src/build/`) and the toolchain's rewritten version-stamped objects (the cache's top-level `build/`). That two-level cache layout is bookkeeping the symbol-rewriting buys, and the spec spends real words keeping the two straight; it is the price of doing versioning in the linker's namespace instead of the source's.

## A tag for humans, a commit for the machine

A version pin itself has to satisfy two readers who want opposite things. Humans want a tag — `v6.2.9` is meaningful, orderable, the thing a changelog talks about. Reproducibility wants a commit hash — immutable, content-addressed, immune to a tag being repointed. Pin only by tag and your build is at the mercy of whoever can move the tag; pin only by commit and the manifest becomes an unreadable wall of hashes nobody can review. Semver *ranges*, the npm/Cargo default, fail the reproducibility test outright: `^6.2.0` resolves to different code on different days, which is the opposite of what a build that has to be bit-reproducible can accept. So Zane records both: the tag in the human-facing manifest, the commit it resolved to in the lock file. The two together are stronger than either alone, and they make a specific attack detectable. When the toolchain fetches, it re-resolves the recorded tag to a current commit and compares it to the recorded one; if they differ, the tag has *moved* since it was added, and the fetch aborts with a security error rather than proceeding. A moved tag is exactly the signature of repository tampering or a hijacked release, and the default stance is to treat it as hostile, not to silently accept the new commit. The cost is friction in the one case where a tag legitimately moves and the user actually wants the new commit — a re-tagged release, say. We do not make that impossible, but we make it *deliberate*: it takes an explicit `--accept-tag-move` override, so trusting a moved tag is always a conscious act with a name, never a silent default. We would rather impose a flag on the rare legitimate move than auto-trust every move and miss the malicious one hiding among them.

## One cache, many spellings of one URL

A verified fetch has to land somewhere on disk, and the somewhere is a single cache shared across every project on the machine ([`dependencies.md` §7](https://github.com/zane-lang/spec/blob/4e54753ce0761896c80126063f115d4dd7c206f7/spec/dependencies.md#7-global-package-cache)) rather than a copy vendored into each project. The per-project alternative is simpler to reason about — every project owns its own dependency tree, nothing is shared, nothing can be corrupted by a neighbor — but it pays for that isolation by refetching and re-rewriting the same `math v6.2.9` once per project that wants it, and by storing the prebuilt objects (which are not small, [chapter two](#shipping-compiled-objects-and-rewriting-their-symbols)) once per project too. Because identity is an immutable URL-plus-commit and the rewrite that produces the version-stamped objects is deterministic, the same version resolves to byte-identical cached artifacts no matter who asks for it, so sharing is safe by construction: there is nothing project-specific in a cache entry to get crossed. A second project that adds the same version reuses the existing `build/` rather than downloading and rewriting again.

Sharing forces the question the per-project model never has to ask: what is the *path* of a cache entry, given that its identity is a URL? A URL is not a filesystem path — it carries a scheme, sometimes a user prefix, a host, and a path, in punctuation a filesystem will not all accept. The first temptation is to hash the URL into a fixed-width opaque directory name, which is collision-resistant and trivially path-safe. We turned that down because an opaque hash makes the cache unbrowsable: a developer who wants to see what is cached, or delete one package, faces a directory of meaningless hex. So the cache mirrors the URL's own structure instead — Go-style mangling, where each `/` of the URL becomes a directory level, so `github.com/zane-lang/math` is three nested, human-legible directories. The price is that the URL must first be reduced to a canonical shape, because the same repository has several legitimate spellings.

That reduction is the part with the most reasoning behind it. One repository can be named `https://github.com/zane-lang/math` or `git@github.com:zane-lang/math` — different scheme, an SSH user prefix, and an SCP-style `:` where the HTTPS form has `/`. Left alone, those two spellings would mangle to two different cache directories and the toolchain would fetch, store, and rewrite the same code twice, and worse, the [multiple-version coexistence](#shipping-compiled-objects-and-rewriting-their-symbols) machinery might see them as two different packages. So normalization strips the scheme, strips the SSH user prefix, and rewrites the SCP `:` to `/`, collapsing both spellings onto the single identity `github.com/zane-lang/math` — one cache entry, one fetch. This is the rare place the design treats two different strings as the same package, and it is deliberate: the strings differ only in *transport*, not in *what is fetched*, so unifying them is recognizing a true equality, not guessing at one.

After normalization there is still punctuation that has no business becoming a directory name — a stray `:`, `@`, `%`, `?`, `#`. Here the design makes a choice it makes nowhere else with paths: rather than escape or percent-encode such characters into something path-safe, `zane add` simply **fails** and refuses the URL ([`dependencies.md` §7](https://github.com/zane-lang/spec/blob/4e54753ce0761896c80126063f115d4dd7c206f7/spec/dependencies.md#7-global-package-cache)). The reasoning is that an escaping scheme is a second, lossy namespace layered over the first — two distinct URLs can escape to the same directory, or one URL can be escaped two ways — and every such scheme eventually leaks. A package URL that needs escaping to be a path is almost always a malformed or hostile input, not a real repository anyone hosts, so the honest move is to reject it loudly at the one moment a human is running a command, not to invent an encoding to swallow it silently. The check runs only over the host-and-path remainder, since the scheme, user prefix, and normalized `:` are exempt by construction — they were already handled. This same forbid-rather-than-escape stance is why `%` is banned in version tags back in [the symbol-rewriting chapter](#shipping-compiled-objects-and-rewriting-their-symbols): the cache path and the symbol separator are screened by one rule for one reason.

The cost the cache layout carries is a two-level structure that the spec spends real words keeping straight, and which exists entirely because of the prebuilt-object decision. A cache entry's `src/` holds the whole cloned repository — including *its* own `build/`, so the author's original `!`-prefixed objects sit at `src/build/` — while the entry's top-level `build/` holds the toolchain's rewritten, version-stamped objects. Two directories both called `build`, at different depths, meaning different things: the author's input and the toolchain's output. That is genuinely confusing, and it is the bookkeeping price of doing version-stamping as a rewrite of shipped binaries rather than at compile time; a source-only model would have one `build/` and no rewrite, at the costs the [shipping chapter](#shipping-compiled-objects-and-rewriting-their-symbols) already accepted.

## No cycles between packages, every freedom within one

Reading a package's manifest pulls in its dependencies, and reading *their* manifests pulls in theirs — fetching is recursive ([`dependencies.md` §9](https://github.com/zane-lang/spec/blob/4e54753ce0761896c80126063f115d4dd7c206f7/spec/dependencies.md#9-transitive-dependencies)), so the dependency relation forms a graph, and a graph raises the question of cycles. Zane forbids them across package boundaries: the package graph **MUST** be a DAG ([`dependencies.md` §10](https://github.com/zane-lang/spec/blob/4e54753ce0761896c80126063f115d4dd7c206f7/spec/dependencies.md#10-package-dependency-graph)), so if `A` depends on `B`, no chain of dependencies may lead from `B` back to `A`. The alternative — tolerating cycles and resolving them, the way a linker resolves mutually-referential object files — was available, but a dependency cycle is almost never a design anyone intended: it means two packages have no clean layering, that neither is really "below" the other, and that they probably want to be one package. Allowing the cycle would let that latent design error compound silently across releases; rejecting it at build time, with an error that names the cycle, turns it into feedback at the moment it is introduced.

The rejection has to be drawn at the right boundary, though, because *within* a single package mutual reference is not an error at all — it is ordinary code. Two functions in different files of the same package calling each other, a type in one file referenced by a function in another: a package is compiled as one unit, so these are just internal references, and forbidding them would be absurd. So the acyclicity rule is scoped strictly to the *package-level* graph and says nothing about intra-package references ([`dependencies.md` §10.1](https://github.com/zane-lang/spec/blob/4e54753ce0761896c80126063f115d4dd7c206f7/spec/dependencies.md#101-single-package-mutual-references)). The line is between units of distribution, not between declarations. The cost is that two packages which genuinely want to reference each other cyclically cannot, and their author is pushed to either merge them or factor the shared part into a third package they both depend on — which is, not by accident, exactly the restructuring the rule is trying to provoke.

## Opt-in remapping

Side-by-side coexistence is safe and simple, but it has a cost that grows with the dependency graph: a program that pulls `math` at `v6.2.9`, `v6.3.1`, and `v6.3.4` through three different paths links three nearly-identical copies of `math` into the binary, even though for most libraries those three are perfectly interchangeable. We wanted a way to collapse that waste — *without* reintroducing the version solver we had just spent the coexistence design avoiding, and without ever silently picking a version that turns out to be incompatible. That tension is the whole of what follows. The dangerous shortcut is the one every range-based ecosystem takes: let a solver decide that `v6.2.9` and `v6.3.4` are "compatible" by semver convention and substitute one for the other. The problem is that Zane links *prebuilt objects*, so substituting one version's compiled code for another's caller is an **ABI** claim — identical layouts, signatures, calling conventions — which is a far stronger and more fragile promise than the source-level "compatible" semver pretends to certify. A wrong guess here is not a compile error you can fix; it is silent undefined behavior at the call boundary, with no recompilation to catch it. Auto-substituting on a version-number heuristic would be handing out that footgun by default.

The resolution splits the decision in two along the line of *who actually knows what*. Only the package **author** knows the real ABI-compatibility windows of their own releases — so the author publishes that knowledge as a `version-pattern` in their manifest. But the author must not be able to *force* substitution onto a consumer, because only the **consumer** bears the consequence if the claim is wrong in their particular link. So the pattern is **information, not permission**: it states which of the author's own versions are interchangeable, and nothing more. The consumer separately, explicitly opts a package in via the top-level `remaps` block. Remapping happens only when the author's facts *and* the consumer's decision agree; absent either, the safe default — coexistence — stands. This is the single most important shape in the document: compatibility facts and the decision to trust them are different jobs held by different people, and the design refuses to let either party unilaterally impose the merge.

Three smaller choices fall out of that frame, each with its own small fork. The `version-pattern` language marks each version component as a fixed boundary (`*`, must match — typically the major), or directionally substitutable (`+` upward, `-` downward), with priority encoded by repetition (`+` outranks `++`) so a pattern is self-describing with no positional convention to memorize; we deliberately did *not* mandate that a pattern contain a `*`, nor that it contain a `+`/`-`, keeping it fully general so authors can express custom schemes, on the reasoning that an over-broad window is the same class of author error as any other and is already caught by the consumer's opt-in and the coexistence fallback (and an all-`*` pattern is harmless on its own terms: it makes only *identical* versions interchangeable, which is a deterministic no-op, not an ambiguous selection). The `remaps` block lists **URLs, not keys** — and this is the one place the design deliberately steps away from the key-nickname convention it uses everywhere else. A key is a local nickname defined in *this* project's `deps`, but remapping's whole point is to collapse versions of a package that may appear *only transitively*, several levels down, and such a package has no key in the top-level project at all — there is nowhere to have defined one. We tried to keep keys here and could not make it cohere: a transitive package's key, if it had one, would live in some intermediate manifest under that intermediate's private naming, and two intermediates can nickname different URLs with the same key or the same URL with different keys, so a key in the top-level `remaps` would be ambiguous exactly when it mattered. The URL is the one identifier that is globally unambiguous and available for *any* node in the resolved graph, direct or not, so `remaps` names packages the canonical way even though the rest of the manifest uses nicknames. And the block is a **bare single-column list** rather than a keyed table, because that is simply what a one-column array *is* in `coda`; a URL is a valid bare token (it contains none of `coda`'s syntax characters), so no quoting or header is needed. A listed URL needs no version pin, since the versions come from the resolved graph.

The cost is the one the whole design is organized around and cannot remove, only contain: a `version-pattern` is an unchecked ABI assertion, and if an author declares a wider window than their ABI actually honors, an opted-in consumer gets silent UB. We do not solve this — there is no ABI verifier here — we *bound* it: remapping is never on by default, it requires a deliberate per-package opt-in, and it always degrades back to safe coexistence when interchangeability cannot be shown. The feature buys smaller binaries at the price of a trust the consumer takes on with their eyes open, and the entire author/consumer split exists to make sure that trust is never taken on by accident. One lesser cost rides along: when two versions of a package declare *different* patterns, full dedup is impossible, and the toolchain says so — but only as a one-time informational note during dependency resolution, not on every build, since a downstream consumer usually cannot fix an upstream pattern mismatch and drowning every compile in the warning would just train people to ignore it.

## The toolchain rides one tag; the standard library does not

A language also has to say how the compiler and its core library are versioned together, and there is a tempting "everything is a package" purity that says the compiler, the core runtime, and the whole standard library should each be independently versioned dependencies. We took a deliberately impure middle line: the compiler and the `core` package are released *together* under a single `zane-version` tag, while everything else in the standard library — `std` and friends — is an ordinary package versioned like any third-party dependency. The reason for coupling compiler and `core` is that the two are genuinely inseparable: `core` is the part of the library the compiler emits calls *into*, so a compiler build and its `core` are co-designed and there is no meaningful sense in which you mix compiler version X with `core` version Y. Pinning them to one tag means a project always builds against a known, coherent toolchain, which in turn frees the toolchain to evolve *without* maintaining backward-compatibility guarantees across its own versions — each project simply states the toolchain it builds with, the same captured-intent move the rest of the language makes. The reason for *not* extending that privilege to the rest of `std` is the opposite: there is nothing inseparable about a hashing library or a JSON parser, so blessing them with toolchain-coupled versioning would just deny them the normal independent-release cadence every other package gets, for no compatibility benefit. So only `core` rides the tag; `std` is fetched, versioned, pinned, and even remapped exactly like any URL-identified dependency, reserved `zane` key and all. The cost is that the toolchain tag is coarse: you cannot upgrade the compiler while holding `core` fixed, or vice versa, because they share one version by construction. We think that is honest rather than limiting — they are not actually separable things — but it does mean a compiler bump is always a `core` bump, and a project that wanted only one of those cannot have it.

## Cutting core loose from the toolchain

That coupling bought coherence, but it did so by making `core` the one exception to a rule the rest of the design holds without apology: every package is fetched at a URL, versioned on its own, added to a manifest, imported by name, and reached by `packageName$` qualification. `core` sat outside all of that — it rode in with the compiler, needed no `deps` row, and its surface types answered to no import. We came to see that exception as the expensive part, not the coherence. The inseparability argument was real but it proved less than it claimed: what actually makes a Zane build coherent is not that the compiler and its library ship under *one tag*, it is that a project pins *exact versions* of everything it builds against — the same captured-intent move [the tag-and-commit pinning](#a-tag-for-humans-a-commit-for-the-machine) makes for every other dependency. A project that names its compiler version and, separately, its `core` version has pinned both just as exactly as one bundled tag would have; the build is no less reproducible for the versions being written on two lines instead of one. So the coherence the coupling was defending never depended on the coupling. Once we saw that, the special standing was doing nothing but carving `core` out of the uniform package rules, and we cut it loose: [`dependencies.md` §14](https://github.com/zane-lang/spec/blob/88b22f00189d910265310b922fffef0b02b8e0e3/spec/dependencies.md#14-toolchain-version) now versions the compiler alone, and `core` is an ordinary package — a `deps` row, a pinned commit, an explicit `import core`, `core$Int` — with no privilege the design denies anything else.

We weighed dissolving `core` the other way, into the compiler, so `Int` and `String` became free-standing language built-ins belonging to no package at all. That removes the exception too, but in the wrong direction: it hard-wires the surface library into the compiler, and it still leaves `Int` reachable by a rule — "it's just there" — that applies to nothing else in the language. It also fights how people already think, which is that `Int` comes *from* `core`; a programmer reaching for it prefers to say so. Making `core` a real package keeps that mental model intact and pays for it with one honest cost, spelled out where the surface types are introduced ([`syntax.md` §2.1](https://github.com/zane-lang/spec/blob/88b22f00189d910265310b922fffef0b02b8e0e3/spec/syntax.md#21-core-surface-types)): a project that never installs `core` cannot name `Int` at all. We take that trade. The types earn their place by being imported, not by being blessed.

The deeper reason to prefer this shape is a feature we have not built yet but mean to: an omni-import, the `from numpy import *` of Zane, that pulls a package's names in unqualified in one file. The whole point of such a thing is that it is *uniform* — one rule that works the same on `math`, on `std`, on anything. A `core` that was a compiler built-in, or a blessed always-present package, would be the single carve-out that rule had to apologize for; unqualified `Int` would come from a different mechanism than unqualified anything-else, and there would be two stories to learn instead of one. By making `core` ordinary now, we make sure the ergonomic path to bare `Int` — when it lands — is the very same path that reaches every other package's names ([`packages.md` §3.1](https://github.com/zane-lang/spec/blob/88b22f00189d910265310b922fffef0b02b8e0e3/spec/packages.md#31-imports-are-file-scoped-namespace-availability) fixes the rule that keeps them qualified until then). This is why the spec's examples already write `Int` bare and elide the `import core`: they are written against the world where that import, plus the omni-import, sits at the top of the file out of frame, not against a world where `core` is magic.

The cost we accept is the one the [previous chapter](#the-toolchain-rides-one-tag-the-standard-library-does-not) named as the reason to couple in the first place, now turned face-up: a compiler and a `core` are versioned independently, so a project *can* pin a `core` that does not match what its compiler emits calls into, a mismatch the single tag made impossible by construction. We do not make that impossible any more; we make it *ordinary* — a version incompatibility like any other, caught, if it is caught, by the same pinning and compatibility machinery that governs every dependency, not prevented outright by a release process. That is a genuine guard the coupling gave for free and this design gives up. We judge it the right price: the alternative was a permanent exception to the package model, and an exception that has to be re-explained at every feature — imports, versioning, and the omni-import still to come — is a heavier long-run tax than a version mismatch that a project pins its way out of.
