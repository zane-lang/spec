# Stories: Lexical Rules

> **See also:** [`spec/lexical.md`](../spec/lexical.md) — the rules these chapters explain.

## Casing carries the kind

Every identifier in Zane answers one question before it answers any other: is this a type or a value? The [foundations story](foundations.md#the-compile-time-model-staged-types-cased-names) tells why that answer is carried in *case* — why an uppercase-initial name is a type and a lowercase one a value, and why the two roads not taken (sigils on every use, or inferring kind from where a name was declared) both lose to a signal every identifier already carries for free. That argument is the language-wide one, and it is not repeated here. What belongs to the lexical layer is what the rule costs and buys once it is in force.

It is enforced as a *hard error*, not a convention or a lint. A lowercase name in a type position is rejected outright ([`lexical.md` §3.1](https://github.com/zane-lang/spec/blob/b5f8af74ed636926ad75c3e6618c785112e137fb/spec/lexical.md#31-types-must-be-uppercase)); there is no path on which the compiler shrugs and guesses. This matters because the parser and every later reader lean on casing as ground truth — a rule that held only most of the time would be a rule neither could build on, the same reason the strictness elsewhere in the language forbids rather than merely discourages. Casing pays off only if it is total.

What it buys, at this layer, is that `<>` can mean type application at all. The angle brackets are also the comparison operators, and the two are told apart with no lookahead and no symbol table: the operand to the left of a `<` is a type exactly when it is uppercase, so `Vector<Int>` opens a type-argument list and `a < b` is a comparison, decided by a single token's first letter ([`lexical.md` §5](https://github.com/zane-lang/spec/blob/b5f8af74ed636926ad75c3e6618c785112e137fb/spec/lexical.md#5-how-casing-disambiguates-the-grammar)). A grammar that had to know what `a` and `Vector` *were* — by remembering their declarations — could not make that call while scanning; casing lets it. The cost is the naming latitude every capitalised-constant or lowercase-type-alias convention from other languages spends freely and Zane cannot; that trade is the foundations story's to weigh, and it does.

## Privacy lives in the name

Casing put a name's *kind* into the name itself, legible at every use site without chasing a declaration. When we came to field privacy, the question had the same shape — where should the fact that a field is private be written? — and it got the same answer, for the same reason.

Two decisions were tangled here, and it helps to take them in turn. The first was *how* privacy is spelled. Languages reach it three ways: a specifier per field (C#'s `private` on each member), a specifier over a range (C++'s `private:` heading a block), or the name itself (Dart's leading underscore). We ruled out the per-field specifier first: it clutters every member with a modifier, and while it is unarguably explicit, explicitness is not its exclusive property — a convention can be just as unambiguous without the noise. C++'s range form we actually liked; it is terse and it groups. But the thing that settled it for the name-based form is a virtue neither specifier form has: the marker rides along to the *use site*. A reader who meets `_id` in an expression sees that it is private right there, with no jump back to the declaration to find out; the specifier forms announce visibility only where the field is defined, and everywhere it is *used* the information is gone. Casing had already committed the language to putting a name's meaning where the name is read, not only where it is written, and privacy is that same idea applied to a different property.

The name-based form wins on a second count too, and it is the one we weigh most heavily: it is the simplest and most minimal of the three. It adds no grammar at all — no `private` keyword, no `private:` section, no per-member modifier slot — because it reuses the identifier the field already has. A property that costs zero new syntax and still reads at every use site is not a close call; the pull of the alternatives is mostly the gravity of the familiar.

So a leading `_` marks a field private ([`lexical.md` §4.2](https://github.com/zane-lang/spec/blob/b5f8af74ed636926ad75c3e6618c785112e137fb/spec/lexical.md#42-leading-underscore)), and it changes nothing about the name's lexical class — the first *letter* still fixes type-versus-value; the underscore is a separate signal layered on top.

The second decision was *what* privacy is private to, and here Zane's extensibility forced a question C++ never has to ask. The model we started from is C++'s: a private field is visible to the type's own methods and hidden from everyone else — privacy is a wall around the type, not around a file or a package. But in Zane methods are not sealed inside the type's declaration; they are package-scope verbs, and a method taking a `this` receiver for some type may be declared in *any* package, including one the type's author never saw. So "the type's own methods" no longer names a closed set. We had to decide whether a `this`-receiver method declared in a foreign package sees the private fields, and we said yes: privacy in Zane is the line between a method and a non-method, not between one package and another ([`types.md` §2.3](https://github.com/zane-lang/spec/blob/b5f8af74ed636926ad75c3e6618c785112e137fb/spec/types.md#23-field-visibility-is-name-based)). Any method whose first parameter is `this` for the type gets in, wherever it lives. We chose the general rule over the guarded one because generality is a bet the whole language makes — a rule with fewer special cases is easier to hold in the head and harder to trip over — and drawing the privacy line at the package boundary would have carved a special case into a model that did not otherwise need one.

The cost is a single, honest one, and it lands in tooling rather than in the language. Because visibility is spelled *in the name*, flipping a field from private to public is a rename, and a rename touches every site that mentions the field, where a specifier-based language flips one token in one place. By hand that is tedious; the real answer is a rename refactor, which is to say the cost is a dependence on a language server that does not exist yet and one day will. Beyond that the decision costs essentially nothing — no runtime, no grammar, no reader confusion — and pretending otherwise to look even-handed would be dishonest.

## A delimiter for each separated thing

Casing and the underscore load meaning onto the identifier; the last lexical decisions load it onto the punctuation *between* identifiers, under one governing rule — no mark may mean two things in the same context, so the parser never needs lookahead to know what a separator is separating.

The rule we settled on is that the delimiter is chosen by *what is being separated* and the bracket by *what kind of thing is inside* ([`lexical.md` §6](https://github.com/zane-lang/spec/blob/b5f8af74ed636926ad75c3e6618c785112e137fb/spec/lexical.md#6-delimiters-and-brackets)). A `;` terminates every member of a `struct` or `variant` body; a `,` separates the elements of a value collection; a newline separates statements; `{ }` encloses a member body or a code block; `[ ]` encloses a flat list. Each character has exactly one job in exactly one setting, so meeting it is unambiguous.

Two of these carry a deliberate asymmetry worth naming. The `;` is *always trailing* — every member ends in one, single-member or many, inline or spread over lines — and the `,` is *never trailing* — it appears only between elements, never after the last. That was a choice against the more common style where a body's last member may drop or optionally keep its separator. Making the terminator unconditional means newlines can be insignificant inside a body without a special rule for the final line, and making the two characters disagree on trailing behaviour keeps their roles visually distinct: a `;` seen after every member reads as *termination*, a `,` seen only between elements reads as *separation*. The bracket split does the same work one level up — because `{ }` always wraps a member body or code block and `[ ]` always wraps a flat list, the bracket alone tells the parser whether the newlines inside it are structural.

The honest cost here is small, and it is worth saying so plainly rather than inventing a drawback. The one thing a newcomer stumbles on is the always-trailing `;`: an inline single-member body still ends in a semicolon — `struct { r Int; g Int; b Int; }` — which looks redundant next to languages that drop the last one. We take the uniformity, one rule with no exception for the final member, over saving a character, because the exception is what would have cost a reader more than the semicolon does.
