# Stories: Effect Model

> **See also:** [`spec/effects.md`](../spec/effects.md) — the rules these chapters explain.

## Inferring effects instead of naming them

Every language that takes effects seriously has to answer one question first: who says what a function does — the author, or the compiler? The well-trodden road is to make the author say it, in a vocabulary built for the purpose. Haskell threads effects through the type as monads; Koka and the effect-system languages put an effect row on every arrow; even the modest version, a `pure`/`readonly`/`io` keyword the programmer sprinkles on declarations, is the same move — the signature carries a hand-written claim about side effects, and the compiler's job is to check the claim rather than discover the fact. We turned that whole family down. An effect annotation is exactly the kind of thing the [foundations bet](foundations.md#the-bet-on-captured-intent) argues a good language should not need the programmer to write: it is a proof obligation bolted onto the signature, a second language to learn beside the first, and — worst for a language that prizes a rule you can read — it can *lie by omission or staleness*, drifting out of step with the body it describes until the annotation says one thing and the code does another.

The resolution is to let the compiler infer the effect level structurally, from facts it already has, and give the user a single modifier — `mut` — and nothing else ([`effects.md` §1](https://github.com/zane-lang/spec/blob/ce1472f48de58b39c8e9695a89956927c680766c/spec/effects.md#1-overview)). There are no purity keywords to write, no effect rows to thread, no capability qualifiers in signatures. A function's classification falls out of two things the compiler can see without help: what its body does to the ownership graph, and what the functions it calls are classified as ([§3](https://github.com/zane-lang/spec/blob/ce1472f48de58b39c8e9695a89956927c680766c/spec/effects.md#3-inferred-effect-levels)). This is the [verifiability symmetry](foundations.md#the-verifiability-symmetry-and-the-one-gap-in-it) cashed out for effects: an effect is a *local, structural* property — does this body write `this`, does it touch a capability, does it call something that does — and local structural facts are exactly the ones a checker confirms cheaply and a reader confirms by looking. Nothing is gained by making the human transcribe a fact the compiler can read straight off the call graph.

The cost is the honest one for any inference that has to stay sound across a boundary it cannot see through: it must be **conservative at the edges**. When the compiler cannot prove what a callee does — an opaque or not-yet-analyzed function — it has no annotation to fall back on, so it must assume the strongest effect level that keeps the program safe ([§5.4](https://github.com/zane-lang/spec/blob/ce1472f48de58b39c8e9695a89956927c680766c/spec/effects.md#54-unknown-callees-are-conservatively-classified)). An annotation system can let an author *promise* purity across such a boundary; we cannot, because we refused the vocabulary in which that promise would be written. We judged that a good trade: the cases where a programmer would want to assert an effect the compiler can't see are rare, and letting them assert it is exactly the channel through which annotations go stale and lie. Better a compiler that occasionally over-classifies a call it can't see into than a signature that can quietly disagree with its body.

## Where mutation is allowed to reach

Inference settles *how* effects are discovered; it does not settle *what mutation even means* in the language, and that question is where the effect model meets the memory model head-on. The unconstrained answer — any function may write through any reference it holds, the way a C pointer or a Rust `&mut` parameter does — was never really available to us, because it collides with the invariant the [memory story](memory.md#safety-without-a-collector-and-without-lifetimes) is built on. If an arbitrary parameter could be mutated in place, mutation would no longer track ownership; two paths to the same object could both claim the right to write it, and the single-owner discipline that buys deterministic destruction would be back to needing a borrow checker to police aliased writes. So mutation had to be tied to ownership, not to aliasing.

The rule that falls out is that `mut` is **receiver-scoped**: it grants write access to `this` and the subtree `this` owns, and to nothing else ([`effects.md` §2.3](https://github.com/zane-lang/spec/blob/ce1472f48de58b39c8e9695a89956927c680766c/spec/effects.md#23-mut)). Parameters other than the receiver are read-only; a method does not get to reach into one and overwrite it ([§2.4](https://github.com/zane-lang/spec/blob/ce1472f48de58b39c8e9695a89956927c680766c/spec/effects.md#24-parameters-are-not-mutable-by-default)). Mutating *another* object is not forbidden — it is just expressed the only way that keeps ownership honest: you call a `mut` method on that object, with it as the receiver, so the write happens through the one path that owns it. And even a `mut` method is not a licence for arbitrary writes; it may write only within the receiver-owned subtree, never an unrelated parameter ([§4.2](https://github.com/zane-lang/spec/blob/ce1472f48de58b39c8e9695a89956927c680766c/spec/effects.md#42-mut-does-not-authorize-arbitrary-writes)). Structs, which have no stable owned identity to mutate, are updated by returning a replacement value rather than by writing a struct receiver in place — the same closed-value discipline the memory model leans on.

The cost lands on anyone who wanted the imperative convenience of an out-parameter: a function cannot take an object, mutate it, and hand nothing back. The mutation has to be phrased as a call on the object itself, or the new value has to be returned. That is more rigid than `&mut`, and deliberately so — the rigidity is what lets the compiler read "this `mut` call writes the receiver's subtree and nothing beyond it" directly off the call, with no aliasing analysis to decide whether two parameters might secretly be the same place. The effect model and the memory model are paying for each other here: mutation is legible *because* it is ownership-scoped, and ownership stays single *because* mutation cannot escape the subtree.

## No ambient I/O: effects you can see in the structure

Receiver-scoped `mut` covers writes to a program's *own* state, but it says nothing about the other kind of effect — the filesystem, the socket, the clock, the logger, the world outside the program. Here the universal convenience is ambient authority: a global `print`, a module-level logger, an `IO` you can reach from anywhere. It reads beautifully at the call site, and that is precisely the problem. Ambient I/O is an effect the source does *not* show — a function that secretly writes a file looks, in its signature and its caller, identical to one that does not. For a language whose whole wager is that intent should be captured in the structure rather than reconstructed by a runtime, an effect you cannot see in the structure is a contradiction in terms.

So the language has no ambient global I/O at all. External state is reachable only through a **capability** — an ordinary object whose methods model access to some piece of the outside world — and a capability is reachable only if it was explicitly passed in or stored ([`effects.md` §6.1](https://github.com/zane-lang/spec/blob/ce1472f48de58b39c8e9695a89956927c680766c/spec/effects.md#61-capabilities-must-be-passed-or-stored-explicitly)). There is no back door: constructor injection records an explicit ownership path to a capability rather than conjuring ambient authority, and a "context object" that bundles several capabilities is just another ordinary object in the ownership graph — it may cut a parameter list down, but it hides nothing, because the capabilities it carries are still reachable only through visible storage and call structure. The consequence the spec is willing to state outright is that **prop drilling is intentional** ([§6.5](https://github.com/zane-lang/spec/blob/ce1472f48de58b39c8e9695a89956927c680766c/spec/effects.md#65-prop-drilling-is-intentional)): threading a capability down through constructors and method calls is not a wart to be designed away with a global, it is the mechanism by which an effect stays visible in the shape of the program.

The cost is exactly the ergonomic tax that ambient authority exists to dodge, and we are choosing to pay it: more parameters, more wiring, the logger that has to be handed down four call frames to reach the place that logs. We considered the usual softeners — an implicit context, a thread-local — and rejected them on the same ground each time: every one of them buys brevity by making an effect reachable without the source showing it, which is the precise thing the model exists to prevent. The capability that has to be passed is annoying to pass and, for that very reason, impossible to overlook — and that legibility is what makes the inference of the previous chapters trustworthy, since a function's reachable effects are all there in its parameters and fields, with nothing ambient leaking in from the side.

## Four levels, and the line between terminating and merely pure

With mutation and capabilities pinned down, the compiler can see, for any function, the strongest thing it does: nothing observable, a capability read, a capability or receiver write. The question that remains is how finely to *grade* that — and the tempting answer is the coarse one, a single pure/impure split, because two buckets are the least machinery that could distinguish a reorderable function from one that is not. We graded more finely than that, into four levels, because two of the payoffs we wanted draw the line in two different places ([`effects.md` §3](https://github.com/zane-lang/spec/blob/ce1472f48de58b39c8e9695a89956927c680766c/spec/effects.md#3-inferred-effect-levels)). One payoff is automatic parallelism — the [strictness-buys-speed bargain](foundations.md#strictness-is-the-performance-model) spent on effects: code that writes no shared mutable state can be reordered and run in parallel, and that needs only the line between "writes nothing observable" and "writes something." The other payoff is **compile-time evaluation**, and it needs a sharper line, because running a function at compile time is only safe if the function is guaranteed to *finish*.

That second requirement is what splits the top of the ladder in two. **Total Pure** functions depend only on parameters and immutable constants and are guaranteed to terminate for all inputs; **Pure** functions have no side effects either, but carry no proof of termination ([§3.1](https://github.com/zane-lang/spec/blob/ce1472f48de58b39c8e9695a89956927c680766c/spec/effects.md#31-level-1--total-pure), [§3.2](https://github.com/zane-lang/spec/blob/ce1472f48de58b39c8e9695a89956927c680766c/spec/effects.md#32-level-2--pure)). Both are equally safe to parallelize — neither writes anything — so a pure/impure split would happily merge them. But only Total Pure is safe to *evaluate at compile time*, because evaluating a merely-Pure function might never return and hang the compiler. Below them sit Read-Only Impure (reads capabilities, writes nothing) and Write Impure (mutates the receiver or the outside world), which grade the impure half by whether concurrent reads can still compose with the single-writer rule. Four levels, then, not as taxonomy for its own sake but because parallelism and compile-time evaluation each need a cut, and the cuts do not coincide.

The cost is inherited from a theorem rather than chosen: termination is undecidable, so the Total Pure tier can only ever be *conservatively* recognized — the compiler admits a function to it via rules it can actually check, and some genuinely-terminating functions will fail those rules and be classified merely Pure, losing compile-time evaluation they could in principle have had. We took that without much agonizing, because it is the only honest option: the alternative is to either promise compile-time evaluation the compiler cannot guarantee will halt, or to make the author assert termination by hand — another annotation, refused for the reasons the [first chapter](#inferring-effects-instead-of-naming-them) gives. A conservative Total Pure tier is the price of getting compile-time evaluation soundly and without a vocabulary.

## What deliberately is not an effect

The last pressure runs the opposite way from the others. The memory model the effect system is built on top of is *busy* underneath — it allocates, it destroys deterministically when a scope drains, it mints anchors, it can abort and unwind. The danger is that all that machinery leaks upward into the effect classification and drags otherwise-pure code into an impure tier for reasons that have nothing to do with what the programmer wrote. A function that builds a small object, reads two fields, and returns a number is, to any reader, pure — and if the mere fact that it allocated a value forced it out of the pure levels, the classification would have stopped tracking *intent* and started tracking *implementation incident*.

So the model draws an explicit line around what is not an effect. Heap allocation and deterministic destruction are runtime implementation events, not side effects for classification purposes; a function stays in the pure levels unless it also writes owned state or touches a capability ([`effects.md` §7.2](https://github.com/zane-lang/spec/blob/ce1472f48de58b39c8e9695a89956927c680766c/spec/effects.md#72-allocation-and-destruction-do-not-by-themselves-raise-effect-level)). Constructors, by the same logic, may allocate freely yet are not `mut`, because creating a fresh value is not mutating an existing receiver ([§7.1](https://github.com/zane-lang/spec/blob/ce1472f48de58b39c8e9695a89956927c680766c/spec/effects.md#71-constructors-may-allocate-but-are-not-mut)). And abortability is held fully orthogonal: a function's abort type and its effect level are independent axes, so an abortable function may be Total Pure or Write Impure depending only on what *else* it does ([§7.3](https://github.com/zane-lang/spec/blob/ce1472f48de58b39c8e9695a89956927c680766c/spec/effects.md#73-abortability-is-orthogonal)) — the [error-handling model](error-handling.md) and the effect model are kept from contaminating each other on purpose.

This is one of the places where the honest account is that the decision cost almost nothing. There is no expressiveness given up, no pattern forbidden; the rule only *declines* to count machinery the memory and error models already pay for as a second time against the effect tier. If it carries a cost at all it is a conceptual one — a reader has to hold the slightly counterintuitive fact that a function which obviously *does* something at runtime (allocates, destroys, may unwind) can still be Total Pure — and that small surprise is a far better deal than the alternative, in which the effect levels would track the runtime's bookkeeping instead of the program's meaning, and almost nothing would be classifiable as pure at all.
