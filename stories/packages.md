# Stories: Packages

> **See also:** [`spec/packages.md`](../spec/packages.md) — the rules these chapters explain.

## The directory is the package

Packages began with two jobs that were already the same boundary in practice: a namespace and a compilation unit. A directory holds the source that belongs together, and its basename is already the name by which that unit is known. We did not seriously consider a separate mechanism that lets a declaration choose some unrelated package. If `a.zn` sits in the `httpClient` directory, it belongs to `httpClient`; there is no second namespace assignment inside the file that can override where the file actually lives. All files in that directory compile as one order-independent unit, so a declaration in one file can use another without imports, forward declarations, or concern for file order ([`packages.md` §2](https://github.com/zane-lang/spec/blob/5ecd2b674577a514fea2f3be7270c0160d9d1af7/spec/packages.md#2-package-identity-and-compilation)).

That made the written `package httpClient` line look redundant, but its redundancy is the point. Moving or copying a source file is a filesystem operation, often performed outside any language-aware tool. Without a declaration, the file silently changes packages when it changes directories, and several mistakes can follow from that one unnoticed move. Requiring every file to repeat the directory basename turns the line into an assertion: the directory determines membership, while the declaration records where the author expected the file to be. A mismatch makes the move visible immediately instead of allowing its consequences to surface later as unrelated name-resolution or visibility failures. The cost is one line repeated in every file. We accepted that small duplication because a safety check must carry independent information to catch the operation it checks.

The package boundary also became the visibility boundary. Names beginning with `_` are accessible across every file in their package and nowhere outside it; the same lexical marker covers types, aliases, constants, verbs, and every other named declaration, while symbol-named operators remain public ([`lexical.md` §4.2](https://github.com/zane-lang/spec/blob/5ecd2b674577a514fea2f3be7270c0160d9d1af7/spec/lexical.md#42-leading-underscore)). That choice belongs to [the lexical privacy story](lexical.md#privacy-lives-in-the-name), not a second packages rationale: the directory gives privacy its extent, while the name makes the boundary visible.

## A barrier that still joins the name

Once another package enters a file, we wanted every use of it to remain explicit. Importing `math` makes the namespace available in that file; it does not pour `math`'s members into the current namespace. A call stays `math$sqrt(value)`, in the spirit of Go's qualified package access. The extra package name is a small typing cost, and in return every cross-package reference tells the reader where it came from without consulting an import list or resolving collisions between identically named members. The current package is the only place whose members can be written unqualified ([`packages.md` §3](https://github.com/zane-lang/spec/blob/5ecd2b674577a514fea2f3be7270c0160d9d1af7/spec/packages.md#3-imports-and-member-access)).

Qualification still left a syntax question. A separator has to tell the parser, not merely the later name resolver, that its left side is a namespace and its right side is a member. A dot could not carry that fact. In `a.b.c`, the punctuation alone does not say which pieces are fields and which, if any, cross a package boundary. We considered `::`, the familiar namespace separator, but it was visually heavier than the operation deserved and sat too close to Zane's `object:method()` call syntax. A cross-package function such as `math::sqrt()` would make a package boundary look like a doubled method marker.

We chose `$` by looking for a single keyboard character that reads as a connection without disappearing or demanding attention. It forms a clear barrier between the two names, yet its width makes the expression look held together; a thin mark such as `¦` split the pieces without visually joining them. The result gives the parser the category boundary directly: `packageName$member` cannot be mistaken for a field chain or a method call ([`packages.md` §3.4](https://github.com/zane-lang/spec/blob/5ecd2b674577a514fea2f3be7270c0160d9d1af7/spec/packages.md#34--separates-a-package-namespace-from-its-member)). The cost is an unfamiliar separator where other languages reuse `.` or `::`; Zane spends that familiarity to preserve one meaning for each delimiter and to keep the namespace crossing visible.

## State has to be a value

A namespace and compilation unit can easily turn into a singleton object by accident. Permit mutable variables at package scope and any function in the package can read or change time-varying state without receiving it. For Zane that is not merely the usual objection to globals. It would destroy the effect model: a call whose visible inputs contain no mutable receiver or state-bearing parameter could still mutate hidden storage, so the signature would stop carrying the information effect inference relies on.

We therefore did not try to make package mutation safer with initialization-order rules, synchronization, or a special global effect. Those approaches preserve the hidden channel and then build machinery around it. The simpler answer is that mutable state is a value. Put it in a `struct` or reference-typed object and pass or store that value through the ordinary ownership, receiver, and capability mechanisms. Packages may still expose immutable constants and verbs, but no time-varying package slot exists ([`packages.md` §5](https://github.com/zane-lang/spec/blob/5ecd2b674577a514fea2f3be7270c0160d9d1af7/spec/packages.md#5-package-scope-state)).

This rule makes some programs more explicit than a global-state design would. A shared service, cache, or registry needs an object, and code that uses it must receive a path to that object instead of reaching through a package name. That wiring is not accidental ceremony; it is the evidence that lets the effect model see where mutation can happen. Packages organize declarations. Values carry state.
